# v4.1 release workflow review

## Current Workflow

The current release model deliberately separates:

1. original repository development and local annotated tag
2. fresh public repository clone outside the original repository
3. sanitized snapshot export and reviewed public docs subset
4. local public verification and snapshot commit
5. explicit normal main push and named tag push
6. post-push Actions and rendered-document verification

This separation is a safety strength and should remain.

## Repeated Hotfix Cause

Two facts become known at different times:

- the original release candidate must be committed and tagged before the
  public snapshot exists
- public Actions success and its run identifier exist only after a push

If README or RELEASE_HISTORY embeds both facts as mutable current status, the
public snapshot needs a second main-only commit after tagging. Moving the tag is
correctly forbidden, so the tag remains on the pure snapshot while main moves
to the status hotfix.

The original README also retains the pre-release tag until another original
commit changes it. That is accurate at candidate time but stale after the
public release.

## Recommended Status Policy

- Update current public tag wording in the sanitized snapshot before its
  snapshot commit and tag.
- Keep the original repository wording neutral where possible, for example by
  directing readers to public Git tags instead of hard-coding a value that can
  only change after release.
- Do not require an exact Actions run ID in durable public release history.
- Link to the Actions page and require the latest release commit to pass as an
  external verification gate.
- If an explicit run ID is intentionally recorded, accept one main-only status
  hotfix and never move the release tag.

## Safe Automation Candidates

Allowed read-only or local preparation automation:

- compare README and RELEASE_HISTORY against an expected tag
- verify working-tree and package-file cleanliness
- list remote main and candidate tag refs
- verify fast-forward ancestry
- build a reviewed copy plan from a public docs allowlist
- scan the prepared folder for forbidden artifacts and public-safe blockers
- run typecheck/test/build/smoke
- print a console-only checklist result

These tools must default to no writes and produce no generated report artifact.

## Steps That Must Stay Explicit

- original annotated tag creation
- selection of the public snapshot commit
- remote URL review
- `git push origin main`
- `git push origin <one-tag>`
- any main-only status hotfix

The process must not automate:

- force push
- `git push --tags`
- tag move, delete, or overwrite
- original repository remote creation
- original history copy
- upload after a failed scan or test

## Public-Safe Snapshot Principles

- export a tree snapshot, never original `.git` history
- use a fresh folder outside the original repository
- exclude dependencies, builds, test outputs, local data, archives, reports,
  and environment files
- include only reviewed public-safe docs
- keep non-public data-origin details out of public text
- use neutral local-data wording

## Hard Stops

Stop before commit or push when:

- original repository remote is present
- the public folder is dirty before snapshot application
- the remote candidate tag already exists
- fast-forward push is not possible
- force push or tag overwrite would be required
- package or lockfile changes unexpectedly
- a dependency changes without approval
- a forbidden artifact or public-safe blocker is found
- typecheck, tests, build, or smoke fails
- README and RELEASE_HISTORY current status disagree

## Review Verdict

Read-only validation automation is justified. Automatic push or tag mutation is
not justified.

## Implemented Read-Only Preflight

Run the local preflight from the repository root:

```powershell
node scripts/public-release-preflight.mjs `
  --candidate-tag v4.1 `
  --expected-current-public-tag v4.0
```

The command reads Git metadata and tracked public text, then reports:

- working-tree cleanliness
- original repository remote absence
- package and lockfile cleanliness
- tracked forbidden artifact paths
- private path, non-fixture email, secret, and mojibake findings
- README and RELEASE_HISTORY current-tag agreement
- SECURITY reporting-policy presence

It exits non-zero when any gate fails. It does not write files, generate a
report, create or change tags, add or change remotes, create a clone, commit,
or push.

Fixture verification is available without a package-script change:

```powershell
node scripts/test-public-release-preflight.mjs
```

The fixture suite covers one passing state and ten targeted failures: dirty
worktree, remote presence, package diff, tracked artifact, private-path/email,
secret, mojibake, stale README, stale release history, and missing security
policy.

## Preflight Hard Stops

A failed check must be reviewed before release preparation continues. In
particular, do not bypass a failure by changing a tag, adding a remote to the
original repository, copying original history, deleting files automatically,
or using force push.

Reserved example domains and explicitly redacted/test path fixtures are not
treated as personal information. Any new exception must be reviewed as source
code; do not weaken the scan merely to obtain a passing result. Intentional
audit documents with scan-target examples remain outside the public snapshot
allowlist.

Tag creation, sanitized snapshot assembly, normal named pushes, and post-push
Actions verification remain explicit later steps. This implementation does
not automate any of them.

## Closeout Verification

The clean original repository passed the implemented preflight with candidate
`v4.1` and expected current public tag `v4.0`. The fixture suite passed one
positive and ten negative cases. Before/after checks confirmed unchanged HEAD,
status, refs, and remotes.

The executable Git calls are limited to branch/HEAD/status/remote reads,
tracked-file listing, and package diffs. No mutation path was found. Tag
creation, fresh public snapshot construction, named normal pushes, and Actions
verification therefore remain explicit release steps.
