## v4.0 Blocker Checklist

Before a v4.0 release candidate, treat the following as blockers:

- typecheck/test/build/smoke failure
- package or lockfile change without explicit approval
- bundled DB/raw/generated artifact in tracked files
- public wording that exposes specific data-origin details
- Trainer live/RTA/OCR/screen capture/overlay/hotkey/watcher/client integration
- server/API persistence for Trainer state without separate approval
- broken core Trainer session, filter, or mistake review flow
- unreadable or stale README current public tag/status text
- inaccessible core Trainer controls that block keyboard use
- narrow layout that hides or overlaps core Trainer controls
- original repo remote unexpectedly present during release work
- public upload requiring force push or tag overwrite

## Non-Blocking Notes

These are not current blockers but should remain visible:

- Node SQLite may emit an experimental warning during tests.
- Sanitized public history is intentionally shorter than the private local
  development history.
- Historical planning docs can contain old release-planning wording; public
  current-status files must be checked before upload.
- Social proof metrics should not be implied or exaggerated.
