## Investigation Summary

Reviewed surfaces:

- current Trainer session/filter/mistake/statistics implementation in `apps/web/src/App.tsx`
- local history and summary helpers in `apps/web/src/trainerHistory.ts`,
  `apps/web/src/trainerSummary.ts`, and `apps/web/src/trainerOptions.ts`
- Trainer and smoke tests under `apps/web/test` and `tests/smoke`
- `docs/v3.7-plan.md`, `docs/v3.7-design-bundle.md`, and `docs/v3.7-closeout.md`
- `README.md`, `SECURITY.md`, `CONTRIBUTING.md`, and `.github/workflows/ci.yml`
- existing public-readiness and v4.0 checklist notes

Observed gaps:

- Trainer state transitions are implemented, but edge-case audit coverage should
  explicitly cover empty candidate lists, filter changes mid-session, completion
  after candidate-count changes, reset after completion, and repeated storage
  failures.
- localStorage keys are versioned, but there is no single readiness checklist
  tying recent history, mistake history, filters, fallback behavior, and future
  migrations together.
- Accessibility and keyboard coverage is baseline-only. Focus order, accessible
  names, selected state semantics, and destructive action clarity should be
  checked before v4.0.
- Narrow/mobile Trainer layout has smoke coverage through normal UI use, but no
  focused responsive checklist for session cards, filter controls, mistake
  filters, and summary blocks.
- Korean-first UI copy is strong in touched Trainer surfaces, but older planning
  docs and public-status text require a final pre-upload audit so stale status
  wording does not leak into future public snapshots.
- SECURITY and contribution docs are serviceable, but v4.0 readiness should
  explicitly verify that public reporting guidance remains clear and does not
  invite sensitive material into public issues.

## Goals

v3.8 should:

- audit and tighten Trainer session state transitions and edge cases
- consolidate localStorage key/version/migration/fallback expectations
- improve accessibility and keyboard behavior for Trainer controls
- check narrow/mobile Trainer layout and text wrapping
- run a Korean-first copy audit over visible Trainer and public-readiness text
- make error, empty, loading, and fallback states consistent
- reduce test duplication where it obscures coverage intent
- add focused smoke or test coverage for the highest-risk v4.0 readiness gaps
- review SECURITY/public-readiness wording
- produce a concrete blocker/non-blocking checklist for v4.0

## Non-Goals

v3.8 must not include:

- new Trainer server persistence
- account, cloud, or cross-device sync
- new dependencies
- package-lock changes without a separate approval step
- DB writes or production DB write paths
- raw data, raw archive, or generated report workflows
- API route expansion for Trainer state
- live/RTA behavior
- OCR
- screen capture
- overlay
- hotkey workflow
- live watcher
- poker client integration
- public upload, GitHub push, or tag work during implementation steps

## Must

- Add focused Trainer state-transition tests or smoke assertions for empty
  candidate lists, filter changes, completion, reset, and retry paths.
- Define and verify a single localStorage key/version checklist for:
  `ptl.trainer.recent.v1`, `ptl.trainer.mistakes.v1`, and
  `ptl.trainer.filters.v1`.
- Confirm corrupt, partial, unknown-version, unavailable-storage, and
  write-failure fallbacks for touched localStorage paths.
- Improve or verify accessible names and keyboard reachability for Trainer
  session controls, filter save/load/reset controls, mistake filters, and
  destructive local clear controls.
- Check narrow/mobile layout for Trainer filters, session status, problem card,
  mistake filters, and local summary blocks.
- Run a Korean-first copy audit over touched Trainer text and public-readiness
  status text.
- Keep package and dependency files unchanged.
- Keep all Trainer state local to the browser.

## Should

- Add a concise v4.0 readiness checklist that separates blockers from
  non-blocking notes.
- Review SECURITY.md and CONTRIBUTING.md for clear public-safe reporting
  guidance.
- Add smoke-friendly selectors or assertions for accessibility and responsive
  expectations where feasible.
- Make empty/error/fallback copy more consistent if the current wording is
  confusing.
- Document any remaining Node warning or public-history note as non-blocking.

## Could

- Add manual QA notes for mobile widths.
- Add a small helper for localStorage key metadata if it reduces duplication
  without adding dependencies.
- Add optional tests for repeated save/load/reset operations across multiple
  browser-like storage instances.
- Prepare a v4.0 release-readiness gate document after v3.8 implementation.
