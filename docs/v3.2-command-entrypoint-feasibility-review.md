# v3.2 Command Entrypoint Feasibility Review

## Purpose

This review checks whether a copied DB rehearsal command entrypoint can be introduced safely after the v3.1 copied DB rehearsal foundation. It is docs-only. It does not create a CLI entrypoint, add a `package.json` script, write a DB, generate a report JSON file, connect API/UI/runtime import paths, or push to GitHub.

## v3.1 Helper Flow

The v3.1 foundation already defines the pure helper chain that any future command must use:

1. Approval contract
2. Rehearsal guard
3. Rehearsal plan builder
4. Rehearsal report shape
5. Dry-run orchestrator
6. Command output renderer

The command layer should remain thin. It should only parse externally supplied arguments, pass normalized inputs into the pure helpers, render safe text output, and map the helper result to a process exit code in a later approved implementation step.

## Candidate Input Parameters

A future source-level entrypoint would need these inputs:

- `--mode dry-run | copied-db-write-rehearsal`
- `--target-kind COPIED_DB`
- `--target-location RESTORE_TEST | BACKUP_ROOT_REHEARSAL`
- `--target-db <redacted-or-externally-validated-path>`
- `--original-db-sha-before`
- `--original-db-sha-after`
- `--copied-db-sha-before`
- `--approval-token`
- `--approve-copied-db-write-rehearsal`
- `--no-report-file-write`

These are design candidates only. This step does not implement argument parsing or wire `process.argv`.

## Mode Separation

### Dry-run Only

Dry-run mode can be feasible first because it keeps all write flags false:

- production DB write: false
- copied DB actual write: false
- report file write: false
- package script exposure: false

Dry-run mode still must reject production DB targets, repo-local DB targets, unsafe copied DB target locations, private path exposure, raw HRC paths, and missing validation evidence.

### Copied DB Write Rehearsal

Copied DB write rehearsal remains a later, separately approved step. Even if the v3.1 pure helper shape can represent `READY_FOR_COPIED_DB_WRITE_REHEARSAL`, this branch must not perform the actual write until a future step approves the exact restore-test target, rollback procedure, and verification evidence.

Production DB and repo-local DB targets must be rejected even with approval.

## Package Script Feasibility

A source-level entrypoint could be reviewed before a package script is added. The safer sequence is:

1. Docs-only feasibility review.
2. Source-level entrypoint implementation without `package.json` script.
3. Manual invocation rehearsal in restore-test or a local development shell.
4. Separate package script approval gate.

This step only completes item 1.

## Report JSON Policy

Report JSON write remains disabled. The v3.1 renderer returns safe text output only. If report file generation is ever needed, it should be planned as a separate approval step with:

- explicit output path guard
- privacy/path scan
- no raw HRC path exposure
- no private token or email exposure
- clear policy for whether generated reports are committed, backed up, or ignored

## Minimum Feasible v3.2 Scope

The minimum feasible v3.2 implementation scope, if later approved, is:

- a source-level entrypoint without package script wiring
- dry-run only by default
- copied DB write rehearsal still blocked by default
- output through the v3.1 safe renderer
- no report JSON write
- no API/UI/runtime import path connection

## Deferred Scope

These remain deferred:

- package script wiring
- actual copied DB write rehearsal
- rollback rehearsal execution
- report JSON generation
- product import route connection
- API/UI import workflow
- production DB or repo-local DB write

## Feasibility Decision

`FEASIBLE_WITH_RESTRICTIONS`

A future source-level entrypoint can be considered only if it remains disconnected from `package.json`, defaults to dry-run behavior, rejects production and repo-local DB targets, uses the v3.1 helper chain, writes no report file, and exposes only safe redacted output.
