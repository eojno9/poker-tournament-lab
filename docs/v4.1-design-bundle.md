# v4.1 design bundle

## Design Intent

v4.1 improves maintainability without changing the stable user workflow. The
design favors explicit module ownership and narrow reusable primitives over a
new framework or a broad abstraction layer.

## Component Boundary

### Application shell

`App.tsx` keeps only cross-screen ownership:

- active tab
- Database-to-Analyze handoff payload
- top-level navigation
- rendering the selected view

The shell must not absorb Trainer, Browser, or storage domain state.

### Trainer boundary

Move `TrainerView` and Trainer-only rendering helpers to one focused module or
small folder. Trainer continues to own:

- solution loading for practice
- filters, hand, and seed inputs
- current problem, answer lock, grade, and cursor
- session counters and completion/reset flow
- recent and mistake history state
- mistake status filter and local summary display

The initial extraction should require no new props. Existing helper modules
remain the source for history, options, and summary logic. DOM structure,
accessible names, test IDs, and copy stay stable.

### Later boundaries

- HRC artifact view is the preferred second extraction because it already has
  sanitized error handling and local state.
- Database can follow after its Analyze handoff contract is exported as a small
  public type.
- Analyze and Browser are deferred because their adjacent helper clusters and
  state surfaces are substantially larger.

## Storage Helper Boundary

Add a small browser-storage module with no domain knowledge:

```ts
interface StorageLike {
  getItem(key: string): string | null;
  setItem(key: string, value: string): void;
}

type StorageReadResult =
  | { ok: true; value: string | null }
  | { ok: false };
```

Candidate operations:

- resolve browser storage or return `null`
- safe text read
- safe text write returning boolean
- safe replacement with a domain-provided fallback string

The helper must not:

- know storage keys
- parse domain payloads
- normalize presets, recent analyses, Trainer entries, or filters
- rename keys or bump versions
- clear unrelated records

Each domain module retains its own compatibility and shape validation. This
keeps migration behavior reviewable and avoids one permissive generic parser.

## API Error Boundary

`api.ts` should throw a narrow error object rather than raw response text. The
error may retain an HTTP status and stable internal category, but its default
message must be public-safe.

Recommended categories:

- network unavailable
- invalid request
- requested item unavailable
- server temporarily unavailable
- unexpected response

A user-message mapper returns concise Korean-first text and a next action. Raw
response bodies, local paths, stack traces, and payloads are not rendered or
logged by default.

The HRC artifact sanitizer remains appropriate for artifact-derived display
text. It should not be replaced by a weaker global string replacement.

## Common State Copy

Do not introduce a design system. A shared status component is allowed only if
it has a stable semantic contract such as:

- tone
- Korean title
- Korean explanation
- optional next-action control
- accessible status role where appropriate

Domain-specific empty, not-solved, invalid, and unavailable meanings should
remain close to their screens.

## Smoke Maintenance

Preserve the seven current scenarios. Maintenance may:

- rename `v1.2 smoke` to a stable workflow suite name
- extract common API route setup
- separate one over-broad scenario by user intent if the total coverage is not
  reduced
- keep public-safety and no-auto-analysis assertions visible

Tests should assert user behavior and stable accessible names, not component
file locations or internal hook structure.

## Release Preflight Design

The release status preflight is read-only and accepts an explicit expected tag.
It may verify:

- README current public tag/status wording
- RELEASE_HISTORY current public tag/status wording
- stale pre-release phrases in current-status sections
- package/package-lock cleanliness
- tracked forbidden artifact absence
- original repo remote absence when run in the original repository

It must not:

- rewrite README or RELEASE_HISTORY
- create a snapshot or report artifact
- add a remote
- create, move, delete, or push a tag
- push a branch
- force push

Exact post-push Actions run IDs should not be required in durable public status
text. Their external timing is the main reason a post-release status hotfix is
otherwise unavoidable.

## Korean-First And Accessibility

- New user-facing error and recovery copy is Korean-first.
- Necessary domain labels such as EV and BB may remain.
- Existing labels, `aria-pressed`, focus-visible, disabled, and heading
  contracts remain stable through extraction.
- Color is not the only status signal.
- Narrow-screen behavior is checked for the extracted Trainer boundary.

## Regression Boundaries

v4.1 must preserve:

- Database-to-Analyze handoff and manual Analyze execution
- Analyze presets and recent history
- Trainer sessions, filters, mistakes, local statistics, and reset scope
- all existing storage keys and legacy migration
- exact/fallback/not-solved distinctions
- read-only artifact review
- off-table and no-client-integration safety copy
