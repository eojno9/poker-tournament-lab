# v4.0 release tag planning

## Scope

This document plans the v4.0 local tag and sanitized public release flow.

This step is docs-only. It does not create `v4.0`, push to GitHub, or push a
tag.

## Release Basis

v4.0 is the stable public milestone for the current off-table study workflow.
It confirms that Analyze, Database, Browser review, Trainer, local verification,
public docs, and sanitized release mechanics work together.

The v4.0 release must remain within the established public-safe scope:

- off-table study only
- no RTA or live assistance
- no OCR, screen capture, overlay, hotkey, watcher, or poker-client integration
- no server-side user history, account sync, or cloud sync
- no bundled DB, raw archive, or generated data/report artifact
- users configure their own local data

## Original Local Tag Plan

After this docs-only bundle is committed and verified:

1. Confirm original repo working tree is clean.
2. Confirm original repo remote is absent.
3. Confirm local `v4.0` tag does not already exist.
4. Create an annotated local tag:

```powershell
git tag -a v4.0 -m "v4.0 - Stable off-table study milestone"
```

The original repository must not receive a remote and must not be pushed
directly.

## Sanitized Public Snapshot Plan

Use a fresh public upload folder outside the original repo:

1. Fresh clone the public GitHub repo.
2. Export the original `v4.0` tag snapshot with `git archive`.
3. Apply only the sanitized public-safe source snapshot.
4. Exclude private history and forbidden artifacts:
   - `.git` from the original repo
   - `node_modules`
   - `dist`
   - `coverage`
   - `test-results`
   - `playwright-report`
   - `artifacts`
   - `.env`
   - DB/SQLite/WAL/SHM files
   - zip files
   - generated reports
   - internal audit/scan docs that are not public-safe
5. Verify typecheck/test/build/smoke in the public folder.
6. Commit the public snapshot on `main` with a public-safe message.

## Public Tag Plan

Create the public `v4.0` annotated tag on the public sanitized snapshot commit,
not on the original private-history commit.

Before pushing:

- working tree clean
- branch `main`
- remote `v4.0` tag absent
- normal fast-forward push possible
- no force push needed
- no tag conflict

Allowed push sequence:

```powershell
git push origin main
git push origin v4.0
```

Forbidden:

- force push
- `git push --tags`
- tag move
- tag delete
- tag overwrite
- original repo push

## Post-Push Verification

After public upload:

- GitHub `main` points to the public v4.0 snapshot or any later status hotfix.
- Remote `v4.0^{}` points to the public v4.0 snapshot commit.
- GitHub Actions succeeds for the pushed main commit.
- README renders with current public tag `v4.0`.
- RELEASE_HISTORY records v4.0 public release completion.
- Public-safe scan remains clean.
- Bundled DB/raw/generated artifacts remain absent.

If README or RELEASE_HISTORY has stale status text after tagging, create a
text-only hotfix on `main` and normal-push `main` only. Do not move, delete, or
re-push the `v4.0` tag.

## Blocking Conditions

Stop the release if any of these occur:

- force push is required
- remote `v4.0` tag exists or conflicts
- tag move/delete/overwrite would be needed
- package/package-lock changes unexpectedly
- dependency changes are introduced
- DB/raw archive/generated artifact is found
- original repo remote is present
- public-safe scan finds a blocker
- typecheck/test/build/smoke fails
- official action refs cannot be verified
