# v3.3 dry-run command skeleton review

## Scope

This document reviews the future dry-run-only copied DB rehearsal command skeleton.

This is a review-only, docs-only step.

No CLI implementation, package script, dependency change, runtime change, API connection, UI connection, DB access, raw data access, test fixture, or generated artifact is created in this step.

## Safety constraints

- Off-table study only
- No RTA
- No live assistance
- No OCR
- No screen capture
- No overlay
- No hotkey
- No live watcher
- No poker client integration
- No production DB write
- No raw DB access
- No raw zip access
- No generated report artifact commit
- No bundled data artifacts
- Users configure their own local data

## Skeleton purpose

The future command skeleton should only validate a local copied rehearsal configuration and render a dry-run console report.

It must not import, mutate, migrate, write, extract, bundle, or connect to API/UI/runtime flows.

The first implementation should be intentionally small. Its job is to prove the command boundary and fail-closed behavior before any broader workflow or package script is considered.

## Candidate future command

Candidate only, not implemented here:

```bash
npm run copied-db:rehearsal -- --dry-run --config <local-config>
```

The exact command name and package script must be separately approved before implementation.

The first implementation may create a source-level command entry point without adding a package script if that is explicitly approved in a future step.

## Allowed future skeleton behavior

A future implementation may:

- parse arguments
- require explicit `--dry-run`
- parse a local config shape without opening raw data
- classify target as copied/local rehearsal only
- reject production-like targets
- reject raw DB/raw zip paths
- render console-only dry-run output
- return non-zero exit code on unsafe input

## Forbidden future skeleton behavior

A future implementation must not:

- write to any DB
- write to production DB
- import data
- migrate data
- open raw DB files
- open raw zip files
- create generated reports by default
- connect to API routes
- connect to Web UI
- connect to poker clients
- observe live gameplay
- use OCR, screen capture, overlay, hotkey, or live watcher behavior
- provide real-time decision support

## Minimal future test matrix

| Test area                           | Expected behavior                            |
| ----------------------------------- | -------------------------------------------- |
| Missing `--dry-run`                 | fail closed                                  |
| Production-like target              | fail closed                                  |
| Raw DB path                         | fail closed                                  |
| Raw zip path                        | fail closed                                  |
| Write/import/apply flag             | fail closed                                  |
| API/UI mode requested               | fail closed                                  |
| Valid local copied rehearsal config | dry-run report only                          |
| Console output                      | no sensitive source/provider/path disclosure |

The first test set should use synthetic config values only. It must not add DB, raw archive, generated report, or local private path fixtures.

## Implementation readiness

`SKELETON_REVIEW_READY`
