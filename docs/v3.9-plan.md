# v3.9 plan

## Goal

v3.9 is the final stabilization pass before v4.0. It should fix practical
readiness gaps that remain after v3.8 without expanding the product into a new
feature line.

## Non-Goals

v3.9 must not include:

- live assistance or real-time decision support
- OCR, screen capture, overlays, hotkeys, live watchers, or client integration
- server-side user history
- account, login, or cloud sync
- bundled DB/raw/generated artifacts
- a new solver or data-generation engine
- broad UI redesign
- dependency or lockfile changes without separate approval
- public upload, tag, or GitHub push in implementation steps

## Must

1. Public status synchronization
   - Update original README and RELEASE_HISTORY wording so it matches the v3.8
     public release state before the next sanitized upload.
   - Add a lightweight scan/check expectation for stale current-tag wording.

2. Korean-first copy and accessible-name pass for older screens
   - Review Analyze, Browser, Database, and artifact surfaces for English-only
     user-facing labels and accessible names.
   - Keep necessary technical constants such as `EV`, `BB`, `canonical key`,
     `HRC_PRECOMPUTED_DB`, and `NOT_SOLVED` when they are domain/status values,
     but explain them with Korean guidance where useful.

3. Analyze to Database workflow clarity
   - Make Database to Analyze handoff state clearer.
   - Ensure reset/back-to-neutral behavior is obvious.
   - Avoid new persistence or cross-screen hidden state.

4. Status and error UX consistency
   - Normalize copy for data-empty, filter-empty, invalid input, request error,
     not-solved, localStorage failure, and missing-detail states.
   - Prefer calm Korean-first recovery guidance.

5. Focused test coverage
   - Add or update web/smoke tests for the above user-facing gaps.
   - Preserve existing core/server/web/smoke counts unless tests are knowingly
     added.

## Should

- Triage GitHub Actions Node runtime deprecation annotations and update workflow
  actions only if the change is low-risk and can be verified.
- Add narrow-screen smoke or CSS checks for non-Trainer long text and controls.
- Create a consolidated localStorage key inventory covering Analyze presets,
  recent analyses, and Trainer keys.
- Add a public upload checklist item that compares original README/RELEASE
  status against the latest public main and tag.

## Could

- Refine smoke test organization comments without changing behavior.
- Add manual QA notes for keyboard traversal by screen.
- Add more explicit empty-state copy for less-used artifact views if touched.

## v4.0 Handoff

v4.0 should receive:

- no known blocker in major Analyze, Database, Trainer, and public-readiness

- current README and release history status aligned with public refs
