boundaries, API error safety, release preflight coverage, and narrow-screen
regression protection without adding a user-facing product capability.

unchanged. AI study explanation, API credentials, server-side user history,
and release mutation are outside this scope.

## Server error safety

The global Express error handler no longer serializes a thrown exception
message. Its response is fixed and public-safe:

```json
{
  "error": "요청을 처리하지 못했습니다.",
  "code": "INTERNAL_SERVER_ERROR"
}
```

Focused tests confirm that thrown detail, submitted raw markers, and stack text
do not appear in the response. Explicit validation and missing-item responses
preserve their existing HTTP status and safe message while adding:

- `INVALID_REQUEST`
- `NOT_FOUND`
- `INTERNAL_SERVER_ERROR`

The `code` field is additive. Successful API response shapes and existing
route-specific validation data remain unchanged. No persistent debug log or
diagnostic artifact was added.

Verdict: `SERVER_ERROR_SAFETY_PASS`

## Frontend error mapping

The browser client recognizes the allowlisted stable codes before applying its
existing HTTP-status mapping. Unknown future codes fall back to HTTP status.
Network, invalid-request, not-found, not-solved, unavailable, server, invalid
response, and unknown paths retain Korean-first user guidance.

Only the allowlisted `code` is read from a failed JSON response. Server error
text, HTML, raw JSON, stack text, and paths are not rendered. `NOT_FOUND` and
`NOT_SOLVED` remain distinct mappings.

Verdict: `FRONTEND_ERROR_MAPPING_PASS`
