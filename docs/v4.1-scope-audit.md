# v4.1 scope audit

## Scope

This audit checks whether work after the v4.0 stable public milestone is
justified by observed maintenance needs. It does not create a new user-facing
feature line.

Baseline:

- v4.0 public release verdict: `V4_0_PUBLIC_RELEASE_COMPLETE`
- public main after the release-status hotfix:
  `7bd50aebb6133aa44f8cccc152df4a0b7d61950d`
- public v4.0 snapshot/tag target:
  `a8f1edc2c4a3b288e5b9acd39cd6c527a7e32110`
- public Actions run `29396539287`: PASS
- npm audit: zero known vulnerabilities at audit time
- original repository remote: absent

## Evidence Summary

### App structure

`apps/web/src/App.tsx` has 5,507 physical lines, 5,176 non-blank lines, and 144
function declarations. It contains the application shell plus all major views
and many view-specific helpers.

The file already has useful internal component boundaries:

| View | Approximate span | Observation |
| --- | ---: | --- |
| `SolutionBrowserView` | 360 lines | Cohesive Browser state plus many nearby Browser helpers |
| `AnalyzeView` | 683 lines | Large state/handler surface and Database handoff contract |
| `TrainerView` | 731 lines | Mostly self-contained state with dedicated helper modules |
| `HrcArtifactsView` | 176 lines | Self-contained read-only view with sanitization helper |
| `ImportView` | 167 lines | Local report/import state with adjacent detail components |
| `DatabaseView` | 288 lines | Owns filters and passes one Analyze handoff callback |

This is a real maintenance issue, but it does not justify a broad rewrite. The
existing function boundaries make a staged extraction possible. Trainer is the
lowest-risk first extraction because its state remains local and its storage
and summary logic already live in focused modules.

### Storage contracts

The following modules each define or consume a local `StorageLike` contract and
repeat storage resolution or JSON handling:

- `apps/web/src/analyzePresets.ts`
- `apps/web/src/recentAnalyses.ts`
- `apps/web/src/trainerHistory.ts`
- `apps/web/src/trainerOptions.ts`

Domain normalization is appropriately different and should remain in each
module. The shared gap is at the storage boundary:

- `getItem` can throw before current JSON fallback logic runs.
- write failure behavior differs between throwing, returning `false`, or being
  silently ignored.
- storage resolution and basic safe read/write logic are repeated.
- tests cover corrupt JSON and several write failures, but do not consistently
  cover a throwing `getItem` across all domains.

A small shared browser-storage adapter is justified. A storage framework, key
rename, version bump, or shared schema validator is not justified.

### Error handling and public copy

`apps/web/src/api.ts` currently throws the raw HTTP response text for failed
requests. Several views render `caught.message` directly. The HRC artifact view
already sanitizes display text, but Analyze, Browser, Trainer, Import, and
Database do not share the same protection.

This can make error copy inconsistent and can expose internal response detail
to the UI. A narrow API error type and Korean-first user-message mapper are
justified. Detailed server bodies should not become default user-facing text.

### Tests and smoke

The verified v4.0 baseline is healthy:

- core tests: 383
- server tests: 82
- web tests: 119
- Playwright smoke: 7/7
- typecheck and build: PASS

The smoke suite protects the major flows, including navigation, Browser,
Analyze, Trainer, Database-to-Analyze handoff, artifact review, and report
fallback states. The remaining maintenance gap is organization:

- `tests/smoke/app-smoke.spec.ts` is 1,164 lines.
- its suite name is still `v1.2 smoke`.
- the first test covers several screens and many unrelated assertions.

Coverage should be preserved. Renaming the suite and extracting route fixtures
or scenario helpers is useful, but deleting or broadly rewriting tests is not a
v4.1 requirement.

### Accessibility, responsive layout, and TypeScript

The current baseline already includes:

- global `:focus-visible` styles
- accessible names on core controls
- responsive breakpoints at 1,120px and 760px
- narrow-grid fallbacks and long-label wrapping safeguards
- TypeScript `strict`, `noUncheckedIndexedAccess`, and
  `exactOptionalPropertyTypes`

No broad accessibility or responsive redesign is justified. Extracted
components must preserve these contracts, and focused regressions should be
added only where boundaries move.

### Public docs and release workflow

README, CONTRIBUTING, and SECURITY provide a usable onboarding and reporting
baseline. SECURITY gives a public-safe route for requesting private contact.

The original README still records v3.9 as the latest completed public tag while
the actual public release is v4.0. This is the direct result of release facts
being known only after the original tag is fixed and the sanitized public
snapshot is pushed. Recording exact post-push status in versioned public text
also creates a second status-only commit after each release.

Safe automation is justified for read-only validation and snapshot preparation
checks. Automatic push, tag creation, tag overwrite, or original-repository
remote setup is not justified.

### CI, dependencies, and Node SQLite warning

- CI uses `actions/checkout@v7`, `actions/setup-node@v7`, and Node.js 24.
- CI runs npm install, typecheck, tests, build, and Playwright smoke.
- public Actions run `29396539287` passed both jobs.
- npm audit reported zero vulnerabilities.
- `node:sqlite` is imported by the server database layer and a compatibility
  test helper. Node.js can still emit an experimental warning for that API.

The warning is not a release blocker. Suppressing all warnings or replacing the
database implementation would create more risk than value. Re-evaluate when
the upstream API status changes.

## User-Visible Change Decision

v4.1 should not add a product capability. Expected user-visible differences
are limited to safer, more consistent Korean error copy. Existing workflows,
storage keys, local records, navigation, and safety boundaries remain stable.
