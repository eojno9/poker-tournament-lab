# v3.0-rc2 Privacy and Backup Policy Fix

## Purpose

This note records the v3.0-rc2 privacy and backup policy hardening applied after the v3.0-rc1 restore rehearsal.

The fix is intentionally limited to privacy-safe fixtures, documentation, and restore policy clarification. It does not create a tag, backup, source archive, git bundle, restore rehearsal, package script, product import route, DB migration, DB write, raw HRC zip copy, generated artifact JSON, or GitHub push.

## v3.0-rc1 Restore Rehearsal Findings

The v3.0-rc1 backup restored successfully as a source archive and git bundle, and the bundle clone passed install, test, build, and smoke verification.

Two release-readiness issues were identified:

- Strict privacy/path scan found real local user/path tokens in committed source, test fixtures, and documentation.
- Bundle clone DB SHA verification was ambiguous because git bundles contain Git-tracked files only and may not include the untracked runtime SQLite DB.

These findings are treated as rc1 release-candidate failure notes, not product runtime known issues.

## Privacy Hardening

Committed files must not include real local user names, private relay email domains, raw HRC folder fragments, local backup roots, or local project roots.

Neutral fixture policy:

- Real user token -> `sample-user`
- Real private token -> `sample-private-token`
- Private relay email -> `sample@example.test`
- Real user home -> `<sample-user-home>`
- Real project root -> `<local-project-root>`
- Real backup root -> `<local-backup-root>`
- Real external HRC source folder -> `<sample-external-hrc-folder>`
- Real DB path -> `<redacted-db-path>/poker-tournament-lab.db`

Allowed placeholders are intentionally sample-only and must not point to a real filesystem location.

## Backup And Restore Responsibility Split

The v3.0-rc2 restore policy separates source archive and git bundle responsibilities:

- Source archive restore verifies file layout, documentation, privacy/path scan, current DB presence, and DB SHA256.
- Git bundle clone verifies Git-tracked source, tags, checkout behavior, and test/build/smoke for tracked code.
- Git bundles may not include the untracked runtime SQLite DB.
- DB SHA verification is performed from the source archive restore or from an explicit DB injection step inside a restore-test folder.
- If bundle clone verification needs DB-dependent commands, the DB must be injected explicitly into the restore-test clone from the source archive or backup root.
- DB injection must happen only inside a restore-test folder and must not modify the original repo.

## Cleanup Policy

Ignored local artifacts must be excluded from backup/source archives:

- `node_modules`
- `apps/web/dist`
- `*.db-shm`
- `*.db-wal`
- raw HRC zip files
- raw extracted files
- generated `hrc-dry-run-*.json`
- `artifacts/hrc-dry-run-reports`
- restore-test folders

Cleanup is a separate verification step. Do not use repo-root destructive cleanup. Do not use PowerShell `-Include` based repo-wide cleanup.

## Safety Boundaries

The rc2 fix keeps these invariants:

- No production DB write.
- No production DB read.
- No DB schema migration.
- No product import route connection.
- No API/UI import flow change.
- No package script for import preview or dry-run import.
- No report JSON generation.
- No raw HRC zip access, copy, extraction, or commit.
- No generated artifact JSON commit.
- No `artifacts/hrc-dry-run-reports` creation.
- No GitHub push.
