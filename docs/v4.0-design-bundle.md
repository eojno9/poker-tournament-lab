# v4.0 design bundle

## Design Intent

v4.0 should make the existing product feel stable, coherent, and publicly
maintainable. It should not widen the safety scope or introduce new persistence.

## Unified Workflow

Target user flow:

1. Open the app and choose a screen from the main navigation.
2. Review available local study spots in Database or Browser.
3. Fill Analyze from a selected Database context when useful.
4. Validate and run Analyze manually.
5. Review exact, fallback, not-solved, or error result states.
6. Reuse presets and recent analyses.
7. Use Trainer for off-table practice.
8. Review mistakes and local statistics.
9. Use verification docs and release notes to audit safety boundaries.

## Workflow Risks To Close

| Risk | v4.0 Handling |
| --- | --- |
| Stale context after Database to Analyze handoff | Keep visible handoff notice and reset only the handoff context. |
| User expects automatic analysis | Keep copy that says Analyze is not run automatically. |
| Reset deletes unrelated records | Keep reset scopes narrow and documented. |
| Not-solved or fallback looks exact | Preserve explicit labels and conservative copy. |
| Empty states leave no next action | Give Korean-first recovery guidance. |
| Storage payload is corrupt or old | Fall back safely without blocking the app. |

## Screen Design

### Entry And Navigation

- Keep top-level tabs predictable.
- Do not hide existing Analyze, Browser, Trainer, Database, or artifact review
  access.
- Maintain accessible names for controls.

### Database And Browser

- Preserve filter/search behavior.
- Keep technical labels where exact values matter.
- Make empty and unavailable states calm and actionable.
- Do not imply bundled public data artifacts.

### Analyze

- Preserve form, JSON mode, validation, presets, recent analyses, and result
  sections.
- Keep Database handoff visible, editable, and resettable.
- Keep validation and request errors Korean-first.
- Preserve conservative exact/fallback/not-solved distinctions.

### Trainer

- Preserve session, filter save/restore, local mistakes, retry/resolved/dismissed
  states, summaries, and local statistics.
- Repair mojibake fallback copy.
- Keep local-only storage and no server persistence.

### Verification And Docs

- README, SECURITY, CONTRIBUTING, RELEASE_HISTORY, and v4.0 docs should describe
  the stable milestone without overclaiming production readiness.
- Keep public wording neutral: users configure their own local data, and bundled
  data artifacts are not included.

## State Design

Use `docs/v4.0-state-model.md` as the stable state contract. Implementation
should not create a separate global state machine unless a narrow helper is
clearly needed.

## Local Data Design

Use `docs/v4.0-local-data-contract.md` as the storage contract. Avoid key or
version changes unless the implementation finds a real compatibility issue.

## Accessibility And Responsive Baseline

v4.0 implementation should verify:

- controls have labels or accessible names
- selected filter states expose state, such as `aria-pressed`, where used
- keyboard navigation remains possible for tabs, filters, and action buttons
- focus-visible styles remain discoverable
- disabled states are clear
- narrow screens do not hide core controls
- long technical labels wrap safely

## Test Plan

Test updates should be focused:

- smoke coverage for the full workflow path where possible
- unit tests for any storage or copy helper touched
- regression checks for Database to Analyze handoff
- Trainer storage and mistake/filter behavior should remain green
- no tests should require DB/raw/generated artifacts

## Safety Design

No v4.0 work may add:

- live assistance
- OCR or screen capture
- overlays, hotkeys, watchers, or poker-client integration
- server-side user history
- bundled DB/raw/generated artifacts
- source/provider/permission disclosure in public docs
