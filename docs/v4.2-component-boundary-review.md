# v4.2 component boundary review

## Pre-implementation App responsibility

`App.tsx` is approximately 4,624 lines. The top-level `App` shell itself is
small and correctly owns navigation plus the transient Database-to-Analyze
handoff. The file remains large because it also contains these screens and
helper clusters:

| Boundary | Approximate source region | Coupling |
| --- | --- | --- |
| Solution Browser | 262-1085 | High: local selection/filter state and many rendering helpers |
| Analyze | 1086-1768 plus result helpers | High: form, JSON, storage, handoff, API, result adapters |
| HRC artifacts | 2278-2578 | Low: local state, GET-only API, existing pure helpers |
| Import/verification | 2592-3243 | Medium/high: import, reports, health, validation, diff |
| Database | 3244 onward | High: filters, detail models, Browser previews, Analyze handoff |
| Formatting and conversion helpers | lower file region | Mixed |

`TrainerView` is already resolved and must not be merged back into the App
shell.

## Next safe extraction

### HRC artifact view: `v4.2 Must`

Reasons:

- the screen takes no props
- all mutable state is local to the screen
- API dependencies are two GET-only functions
- list, empty, error, detail, and filter states form a coherent boundary
- domain transformations already live in `hrcArtifactDashboard.ts`
- existing smoke coverage protects the read-only and error-sanitization
  contract
- extraction removes roughly 300 lines plus related imports from `App.tsx`

Risk controls:

- preserve test IDs, labels, headings, and visible copy
- preserve initial refresh and manual refresh behavior
- preserve GET-only assertions
- avoid importing any App-local type or component
- do not combine extraction with a visual redesign

Verdict: `HRC_VIEW_EXTRACTION_READY`

## Analyze/Browser/Database boundary

### Analyze full extraction: `v4.3+ backlog`

Analyze owns many related state values and consumes transient handoff props.
Moving the whole screen now would produce a very large diff with little direct
user value. A pure handoff module is a safer v4.2 Should item.

### Browser full extraction: `v4.3+ backlog`

Browser is internally cohesive but includes a large rendering/helper cluster.
It should be moved only after its models and formatting helpers are grouped and
focused component tests exist.

### Database full extraction: `v4.3+ backlog`

Database owns the most important cross-view callbacks. Its extraction should
follow an exported Analyze handoff contract, not precede it.

### Import/verification extraction: `v4.3+ backlog`

Import, report, validation, canonical-diff, and DB-health responsibilities can
eventually become separate panels. They are not a release or safety blocker.

## Helper-first opportunities

`v4.2 Should`:

- exported Analyze handoff types and pure builders
- a neutral detail-row primitive only if needed by HRC extraction
- moving HRC-only formatting imports with the HRC view

Avoid generic state hooks or a global store. Current ownership is explicit and
does not justify a new framework.

## Decision

Extract HRC artifacts in v4.2. Prepare, but do not execute, broader Analyze,
Browser, Database, or Import decomposition.

## Extraction result

The approved HRC boundary is now implemented:

| Item | Before | After |
| --- | --- | --- |
| `App.tsx` | 4,624 lines | 4,312 lines |
| HRC view ownership | embedded in `App.tsx` | `HrcArtifactView.tsx`, 332 lines |
| HRC state and requests | App-local | view-local, GET-only |
| App HRC responsibility | full screen | import and conditional mount only |

The extracted view preserves existing test IDs, accessible names, Korean-first
copy, list/detail filters, empty/error/loading states, and sanitized previews.
It imports API and HRC dashboard helpers directly and does not import from
`App.tsx`, so no reverse or circular dependency was introduced.

The narrow HRC table now uses a constrained single-column grid track and
table-local horizontal scrolling. A 390 px smoke viewport verifies that the
document does not overflow while dense table content remains reachable.

Analyze, Browser, Database, Import, and Database-to-Analyze handoff movement
was not performed. Those boundaries have higher coupling, and changing them in
the same maintenance commit would broaden regression risk without closing an
active blocker.

Implementation verdict: `V4_2_IMPLEMENTATION_BUNDLE_READY`

## Closeout verdict

Regression verification confirms list, empty, error, detail, comparison,
refresh, sanitization, accessibility, and GET-only behavior. The 390 px smoke
path also confirms document-width containment and table-local scrolling.

`HRC_VIEW_BOUNDARY_PASS`
