# v3.3 copied DB command implementation plan

## Scope

This document defines the implementation plan for a future copied DB rehearsal command.

This is a planning-only, docs-only step.

No command implementation, dependency change, runtime change, API connection, UI connection, DB write, raw data access, or generated artifact creation is performed in this step.

## Safety constraints

- Off-table study only
- No RTA
- No live assistance
- No OCR
- No screen capture
- No overlay
- No hotkey
- No live watcher
- No poker client integration
- No production DB write
- No raw DB access
- No raw zip access
- No generated report artifact commit
- No bundled data artifacts
- Users configure their own local data

## Background

The v3.0-v3.2 work established a no-write foundation for local study tooling, copied DB rehearsal planning, safe command output, and sanitized public repository handling.

The next possible implementation layer is a local copied DB rehearsal command. It must remain disconnected from production DB targets, API routes, UI workflows, package scripts, raw archives, and generated report files until those surfaces are separately reviewed and approved.

Public wording must stay neutral:

- locally configured precomputed study data
- external/local data not included
- users configure their own local data
- no bundled data artifacts

This document intentionally does not name any specific dataset, data provider, original file, original path, private backup root, or permission history.

## Proposed command purpose

The future command is a local dry-run rehearsal helper.

It should:

- validate configuration shape
- validate target approval contract
- build a rehearsal plan
- render a dry-run report
- avoid all DB writes
- avoid raw data access
- avoid production DB targets
- avoid API/UI connection

It should not:

- import data
- mutate DB files
- open raw archives
- connect to poker clients
- support live play
- provide real-time decision support

## Proposed command name

Candidate only, not implemented in this step.

Example candidate:

```bash
npm run copied-db:rehearsal -- --dry-run --config <local-config>
```

The exact script name must be finalized in a later implementation step. This step does not add the script to `package.json`.

## Proposed input contract

Candidate fields:

| Field          | Required | Description                              | Safety note                                     |
| -------------- | -------- | ---------------------------------------- | ----------------------------------------------- |
| config path    | yes      | Path to local user-owned config          | Must not point to raw DB/raw zip                |
| dry-run flag   | yes      | Must be explicitly enabled               | Command must fail closed if absent              |
| target kind    | yes      | copied/local rehearsal target only       | Production target forbidden                     |
| approval token | yes      | Explicit local rehearsal approval marker | Must not imply production write approval        |
| output mode    | no       | console-only by default                  | File output disabled unless separately approved |

Additional input constraints:

- Config input must be user-owned and local.
- Config input must not encode private backup roots.
- Config input must not include specific data source/provider names.
- Config input must not include original raw file names or original raw paths.
- Target classification must be derived before any future file access.
- Any ambiguous target must be blocked.

## Proposed output contract

The future command may print a console-only report containing:

| Field                 | Description                                 |
| --------------------- | ------------------------------------------- |
| command mode          | Always dry-run                              |
| target classification | copied/local rehearsal target only          |
| write status          | Always no-write                             |
| raw data status       | Not accessed                                |
| production DB status  | Forbidden / not targeted                    |
| validation summary    | Pass/fail checks                            |
| planned actions       | What would happen in a future approved step |
| blocked actions       | Anything refused by safety gates            |
| final verdict         | `DRY_RUN_PASS` or `DRY_RUN_BLOCKED`         |

No generated report file should be committed.

Output must not print:

- unredacted local absolute paths
- private user identifiers
- raw archive paths
- raw DB paths
- specific data source/provider names
- permission claims
- live-play or client-integration instructions

## No-write safety gate

The future implementation must fail closed unless all conditions are true:

- explicit `--dry-run` flag is present
- target is classified as copied/local rehearsal only
- target is not production
- no write flag is provided
- no import/apply/migrate/commit operation is requested
- no raw DB/raw zip path is provided
- no API/UI/runtime integration is requested

If any condition fails, the command must exit without modifying files.

The implementation must not create sidecar files, report JSON, backup folders, restore-test folders, or generated artifacts unless a later step explicitly changes the policy.

## Target approval contract

The approval contract must approve only a local copied rehearsal target.

The contract must:

- approve only local copied rehearsal targets
- not approve production DB writes
- not approve raw data access
- not approve bundled artifact creation
- not encode private paths
- not include provider/source/dataset names

The approval marker is not a production write approval. It only allows a dry-run command to describe a copied DB rehearsal plan.

## Raw data access prohibition

The future command must reject:

- raw DB file paths
- raw archive paths
- raw data folder paths
- generated report input paths that could reveal raw file names
- config values that require opening raw archive contents

The dry-run command should classify and reject these inputs from strings or metadata only. It must not inspect raw DB or raw archive contents.

## Production DB write prohibition

Production DB targets are always forbidden.

The command must reject production-like targets even if:

- the approval marker is present
- the target path appears local
- the user passes an apply/write/import flag
- the config labels the target as copied but other evidence is ambiguous

The safe default is `DRY_RUN_BLOCKED`.

## Allowed local copied DB conditions

A copied DB rehearsal target can be considered only if all of these are true:

- target kind is explicitly copied/local rehearsal
- target location is outside production DB ownership
- target does not point to raw DB/raw zip data
- target does not encode private backup roots in public output
- no write/import/apply/migrate flag is present
- console output can be redacted safely
- no API/UI/runtime connection is requested

This Step 3 does not approve actual copied DB writes.

## Failure modes

| Failure                          | Expected behavior                            |
| -------------------------------- | -------------------------------------------- |
| missing `--dry-run`              | fail closed                                  |
| production-like target           | fail closed                                  |
| raw DB path provided             | fail closed                                  |
| raw zip path provided            | fail closed                                  |
| write/import/apply flag provided | fail closed                                  |
| API/UI connection requested      | fail closed                                  |
| unsafe config wording            | fail closed or warning depending on severity |
| ambiguous target classification  | fail closed                                  |
| output cannot be safely redacted | fail closed                                  |
