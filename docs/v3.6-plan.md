v3.6 must not include:

- RTA behavior
- live assistance
- OCR
- screen capture
- overlay
- hotkey
- live watcher
- poker client integration
- live game decision support
- bundled DB/raw data/generated reports
- production DB writes
- server-side Trainer history storage
- account/user tracking for Trainer practice history
- package/dependency changes without a separate approval step
- GitHub push or tag operations in planning steps

## Korean-First UI Principles

Future v3.6 implementation should keep the UI Korean-first:

- Primary headings, labels, buttons, feedback, empty states, and help text should be Korean-first.
- Domain terms such as `EV`, `BB`, and canonical identifiers may remain in technical form where translation would reduce precision.
- Safety copy should stay visible but concise.
- Empty, loading, unsupported, and fallback states should be understandable without reading implementation docs.
- The Trainer should feel like an off-table learning tool, not a live-game assistant.

## Safety Constraints

All v3.6 work must keep these constraints:

- Off-table study only.
- No RTA or live assistance.
- No OCR, screen capture, overlay, hotkey, watcher, or poker client integration.
- No DB/raw zip/generated report access.
- No bundled data artifacts.
- Users configure their own local data.
- Trainer state remains local unless separately approved.
- Public wording remains neutral and does not disclose source/provider/permission details.
