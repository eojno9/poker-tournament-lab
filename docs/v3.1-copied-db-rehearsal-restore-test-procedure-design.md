# v3.1 Copied DB Rehearsal Restore-test Procedure Design

## Purpose

This docs-only design defines how a future copied DB rehearsal should be constrained to a restore-test area that is isolated from the original repository and production DB.

The procedure preserves the original production DB SHA before and after any future copied DB rehearsal. This step does not create a restore-test folder, copy a DB, write a DB, run an import, implement a CLI entrypoint, add a package script, or write report JSON.

## Restore-test Location Policy

Restore-test work is allowed only under a repo-external backup root. The procedure must use placeholder-safe paths in public-facing docs:

`<local-backup-root>/restore-test-v3.1-copied-db-rehearsal-YYYYMMDD-HHMMSS`

Target rules:

- production DB target is forbidden
- repo-local DB target is forbidden
- copied DB target is the only candidate
- copied DB target must be under a restore-test or backup-root rehearsal location
- target paths must be reported only through redacted summaries

## Procedure Design

### Step A: Restore-test Workspace Location Check

Confirm the intended restore-test workspace is repo-external and under the approved backup root placeholder.

### Step B: Copied DB Preparation Source

Prepare a copied DB only from a source archive DB or another approved copied DB source. This design step does not perform the copy.

### Step C: Original DB SHA Before

Record the original production DB SHA before any future rehearsal. Expected v3.1 baseline:

`92BBF8DA75EA34ABF4CBCF8111ADBCD729DA93D7DDE690E14EB52482A8E16461`

### Step D: Copied DB SHA Before

Record the copied DB SHA before any future copied DB write rehearsal.

### Step E: Preview Validation

Confirm import preview validation passes before write-capable rehearsal is considered.

### Step F: Duplicate and Canonical Key Validation

Confirm duplicate/canonical key validation passes before write-capable rehearsal is considered.

### Step G: Approval Contract

Confirm explicit approval evidence for copied DB write rehearsal. Dry-run-only mode may proceed only with write flags false.

### Step H: Rehearsal Guard

Confirm the guard accepts only a copied DB target in `RESTORE_TEST` or `BACKUP_ROOT_REHEARSAL`.

### Step I: Plan Builder

Confirm the plan status is either `READY_FOR_DRY_RUN`, `READY_FOR_COPIED_DB_WRITE_REHEARSAL`, or `BLOCKED` with clear reasons.

### Step J: Dry-run Orchestrator

Confirm orchestrator output is generated from in-memory evidence only.

### Step K: Output Renderer

Use the safe output renderer to review console summary only. Do not print or store actual local paths, email addresses, private tokens, raw zip paths, or raw HRC source paths.

### Step L: Rollback Plan

Confirm rollback plan availability before any future copied DB write rehearsal.

### Step M: Cleanup Policy Check

Confirm cleanup scope is limited to the restore-test workspace and explicit paths only.

## Write Rehearsal Gate

This document is design-only. Actual copied DB write rehearsal remains forbidden until a separate approval step.

A future copied DB write rehearsal requires all of the following:

- restore-test or backup-root rehearsal location
- copied DB target only
- explicit approval flag or token
- rollback plan
- original DB SHA evidence
- copied DB SHA evidence
- preview validation pass
- duplicate/canonical validation pass
- privacy/path scan pass
- raw zip absence
- artifact report absence
- product import route disconnected
- API/UI runtime unchanged

Production DB targets are rejected even with approval.

## Rollback Verification Policy

A future rollback verification must record:

- copied DB SHA before rehearsal
- copied DB SHA after rehearsal
- copied DB SHA after rollback
- original DB SHA before rehearsal
- original DB SHA after rehearsal

The original DB SHA before and after must match. If rollback verification fails, v3.1 must not enter rc readiness.

Rollback implementation or rehearsal requires a separate step.

## Cleanup Policy

Cleanup must be explicit and local to the restore-test workspace.

Forbidden cleanup patterns:

- broad cleanup
- PowerShell `-Include` based cleanup
- repo root `Remove-Item -Recurse`
- `.git` deletion or modification
- production DB deletion
- DB body deletion outside the restore-test copied DB target

Allowed cleanup design:

- remove only explicitly named restore-test generated paths
- never target the repo root
- never target `.git`
- never remove the production DB body
- never use broad file patterns that can escape the restore-test workspace

## Output and Report Policy

Default output is safe console summary only.

Report JSON file writing remains disabled until a separate approval step. Any future report output must pass privacy/path scan and must not expose actual local paths, email addresses, private tokens, raw zip paths, or raw HRC source paths.
