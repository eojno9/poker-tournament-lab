## Release content

- extracted GET-only HRC artifact view boundary
- generic public-safe global server errors with additive stable codes
- compatible Korean-first browser error mapping
- explicit original/public read-only release preflight modes
- optional JSON preflight stdout
- focused 390 px HRC overflow regression coverage
- synchronized v4.1 completed/v4.2 local status wording

No new product feature, dependency, storage version, database behavior, API
persistence, or AI explanation capability is part of this candidate.

## Required procedure

1. Run original preflight with `--mode original`.
2. Create one original annotated `v4.2` tag at the final closeout HEAD.
3. Fresh-clone the public repository into a separate upload folder.
4. Export the original tag with `git archive` to a temporary snapshot folder.
5. Apply the approved public allowlist and internal-doc exclusion policy.
6. Copy the sanitized snapshot into the public clone without copying original
   Git history or forbidden artifacts.
7. Run public preflight with `--mode public`.
8. Run `npm ci`, typecheck, tests, build, and smoke.
9. Repeat public-safe, UTF-8, artifact, and status scans.
10. Commit the public snapshot as a normal descendant of current public main.
11. Create one public annotated `v4.2` tag on that snapshot commit.
12. Push `main` normally.
13. Push only the named `v4.2` tag normally.
14. Verify GitHub Actions, remote refs, README, and RELEASE_HISTORY.
15. If durable status text is stale, use a main-only docs hotfix; never move
    the `v4.2` tag.

## Pre-push gates

- working tree clean and branch `main`
- expected public origin only
- remote main unchanged since preflight
- local snapshot is a normal fast-forward descendant
- remote `v4.2` absent
- original history and internal policy/audit documents excluded
- package and lock files unchanged by snapshot preparation
- tracked artifacts and bundled data zero
- privacy/path/email, secret, data-disclosure, and mojibake blockers zero
- README names `v4.1` until the release status is actually updated
- all local verification commands pass
- user approval for normal main and named-tag push remains in force

## Prohibited operations

- force push
- `git push --tags`
- existing tag movement, deletion, overwrite, or recreation
- adding a remote to the original repository
- copying original `.git` history
- package/lock or dependency changes
- storage key/version changes
- bundled DB, raw archive, sidecar, generated report, or build/test artifact
- preflight-driven commit, tag, file mutation, or push

## Status hotfix policy

The release tag identifies the verified public snapshot. A later README or
RELEASE_HISTORY correction may advance only public `main`. It must not retarget
or repush `v4.2`.
