# v3.1 Copied DB Command Entrypoint Design

## Purpose

This docs-only design defines how a future copied DB rehearsal command entrypoint could safely use the v3.1 helper chain:

1. approval contract
2. rehearsal guard
3. rehearsal plan builder
4. rehearsal report shape
5. rehearsal dry-run orchestrator
6. command output renderer

The entrypoint remains hypothetical in this step. No command file, package script, DB copy, DB write, report JSON file, API/UI import flow, or product import route is added.

## Entrypoint Non-goals

A future entrypoint is not:

- a production DB import command
- a repo-local DB write command
- a raw HRC zip import command
- an API/UI product import flow
- a `package.json` script added by default
- a report JSON file writer
- a DB schema migration command
- a production DB backup/restore command

## Candidate Entrypoint Location

Possible future file location:

- `apps/server/src/cli/importHrcCopiedDbRehearsal.ts`

This is an example only. This step does not create that file.

## Input Parameter Design

Future command parameters may include:

- `--mode dry-run | copied-db-write-rehearsal`
- `--target-kind COPIED_DB`
- `--target-location RESTORE_TEST | BACKUP_ROOT_REHEARSAL`
- `--target-db <redacted-or-externally-validated-path>`
- `--original-db-sha-before <sha256>`
- `--original-db-sha-after <sha256>`
- `--copied-db-sha-before <sha256>`
- `--approval-token <token>`
- `--approve-copied-db-write-rehearsal`
- `--no-report-file-write`

These flags are examples only. They are not wired to a command or package script in this step.

## Safety Policy

The future entrypoint must preserve these rules:

- production DB targets are always rejected
- repo-local DB targets are always rejected
- copied DB targets outside `RESTORE_TEST` or `BACKUP_ROOT_REHEARSAL` are rejected
- copied DB write rehearsal requires explicit approval
- dry-run mode is allowed only with write flags false
- report file write is disabled by default
- output is produced only through the Step 9 renderer
- actual local paths, emails, tokens, raw zip paths, and raw HRC source paths must not be printed
- raw HRC zip access remains out of scope
- product import routes remain disconnected
- API/UI import flow remains absent

## Output Renderer Policy

The future entrypoint should pass the Step 6 orchestrator result to the Step 9 output renderer. It should not format raw helper output itself. This keeps redaction and safety display behavior centralized.

The rendered output must show:

- `Can Production DB Write: false`
- `Can Write Report File: false`
- execution evidence for actual DB copy/write/report JSON writes
- blocked reasons and exit code when any safety gate fails

## Exit Code Policy

The future entrypoint should map the helper exit codes directly:

- `0`: ready or success shape
- `1`: validation or plan blocked
- `2`: safety blocked
- `3`: privacy/path blocked
- `4`: input or internal shape blocked

## Implementation Gate

Before an entrypoint implementation is allowed:

- v3.1 interim readiness review must pass
- package script exposure must remain separately approved
- command tests must prove production and repo-local target rejection
- command tests must prove no report file write occurs by default
- command tests must prove output goes through the renderer
- command tests must prove no raw HRC zip access occurs

Before any package script is added:

- exact script name must be documented
- default mode must be dry-run or copied-DB-only
- production DB write must remain impossible
- report JSON file writing must remain separately approved

Before any copied DB write rehearsal:

- restore-test backup/rollback rehearsal policy must exist
- copied DB target must pass path guard checks
- original DB SHA before/after must match
- copied DB SHA baseline must be recorded
- rollback verification must be planned
- privacy scan must pass
- raw zip and artifact report absence must be confirmed

Before any report JSON write:

- a separate report policy must define file path guards
- report file writing must be explicitly approved
- report output must pass privacy/path scanning
- generated reports must not be committed automatically
