# poker-tournament-lab v1.8 Plan

## v1.8 Summary

v1.8 extends the v1.7 `hand -> actions[]` multi-action foundation into an import/storage-ready v2 strategy format. The goal is to preserve action-level frequency and EV data for fold/call/raise/all-in lines, then make that data available to the existing Database and Analyze read-only previews.

This release is not a new solver. It must preserve existing exact lookup, fallback routing, Trainer behavior, and v1.0-v1.7 imported data compatibility.

## Product Goals

- Short term: Import HRC-built DBs accurately and explore/study them like GTO Wizard.
- Mid term: Let users choose only DB-existing actions/raise sizes, then display raise/call/fold/all-in frequencies and EVs clearly.
- Long term: Create calculation jobs like HRC Pro / ICMIZER and keep producing GTO DBs from internal or external calculation results.
- Final goal: Build an off-table MTT preflop/ICM GTO DB creation, exploration, and study platform.

## Goals

- Define a normalized JSON v2 import format for multi-action strategies.

## Workstreams

### A. Multi-Action Import v2 Schema

- Define normalized JSON v2 shape.
- Add `schemaVersion` or equivalent version metadata.
- Support solution records whose `strategy` can contain `hand -> actions[]`.
- Each action can include:
  - `action`: `FOLD`, `CHECK`, `CALL`, `BET`, `RAISE`, `ALL_IN`, or `UNKNOWN`
  - `sizeBb`
  - `sizePctPot`
  - `isAllIn`
  - `rawSizeLabel`
  - `frequency`
  - `ev`
  - `chipEv`
  - `icmEv`
  - `sourceActionLabel`
  - `warnings`
- Keep existing v1 strategy shape readable through the v1.7 legacy adapter.

### B. Strategy v2 Validation

- Validate hand notation.
- Validate that `actions[]` exists and is non-empty for v2 hands.
- Normalize action kinds without inventing missing actions.
- Validate frequency range `0..1`.
- Add warning for suspicious frequency sums rather than silently correcting data.
- Allow EV, ChipEV, and ICM EV to be `number` or `null`.
- Add non-blocking size warnings for `RAISE`, `BET`, and `CALL` when size is not explicit.
- Do not require size for `ALL_IN`.
- Add warning for `UNKNOWN` action.
- Record invalid rows, warning rows, and skipped rows clearly in reports.
- Reuse the same validator for dry-run validation and actual import.

### C. Import/Store Compatibility

- Preserve existing `solutions.strategy` JSON compatibility.
- Store v2 strategy shape only for v2 import records.
- Keep existing v1 imported data unchanged.
- Preserve existing `canonical_key`, source label, and exact lookup behavior.
- If DB migration becomes necessary, split it into a separate phase after read/write compatibility tests.
- Import reports should include `schemaVersion` and `multiActionStrategyCount` when v2 data is present.

### D. `/api/solutions` Read Compatibility

- Keep `/api/solutions` read-compatible with both v1 and v2 strategy payloads.
- Do not add a new API unless v2 import compatibility clearly requires it.
- Preferred direction: keep existing import endpoint and branch on payload `schemaVersion`.
- Return v2 strategy JSON as stored so the web adapter can display native `actions[]`.

### E. Database/Analyze v2 Strategy Display Connection

- Prefer native v2 strategy data when present.
- Continue using the v1.7 legacy adapter for v1 single-action strategy data.
- Database multi-action preview should display native v2 actions without DB schema migration assumptions.
- Analyze multi-action detail should display native v2 actions when strategy data is available.
- Show action-level frequency, EV, ChipEV, ICM EV, size, source label, and warnings.
- Do not build the full GTO Wizard-style Solution Browser v2 in this release.

### F. Tests, Smoke, And Scope Guard

- Add core unit tests for v2 schema validation.
- Add sample v2 JSON fixtures.
- Add import validation tests for v2 valid, invalid, warning, and skipped rows.
- Add compatibility tests proving v1 imports and v1.0-v1.7 data still display.
- Add API/read compatibility tests for `/api/solutions`.
- Extend smoke tests only after v2 data is connected to existing Database/Analyze displays.
- Re-run scope guard before final verification.

## DB/Storage Strategy Comparison

### Option 1. JSON Backward-Compatible Storage

- Store v2 strategy shape in the existing `solutions.strategy` JSON field.
- Keep v1 strategy shape unchanged for existing records.
- Use `schemaVersion` to distinguish v1 and v2 imports.
- Lowest migration risk.
- Best fit for v1.8 because v1.7 already displays `actions[]` through read-only adapters.
- Requires strong validation and read compatibility tests so mixed v1/v2 data remains safe.

### Option 2. DB Migration

- Add normalized action-level tables or columns for native multi-action storage.
- Better long-term querying and indexing potential.
- Higher risk for v1.0-v1.7 compatibility.
- Requires migration, rollback, backup, and compatibility planning.
- Should be deferred unless JSON storage cannot preserve required v2 data safely.

### v1.8 Recommended Strategy

Use JSON backward-compatible storage first. Keep existing `solutions.strategy` JSON readable and allow v2 import records to store native `hand -> actions[]` strategy shape with `schemaVersion`. Defer DB migration to a later release after v2 validation, import reports, read compatibility, and UI display behavior are proven.
