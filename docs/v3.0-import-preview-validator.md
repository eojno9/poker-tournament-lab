# v3.0 HRC Import Preview Validator

## Purpose

This document defines the v3.0 import preview validator that extends the Step 2 preview contract into a duplicate and canonical-key validation pass.

The validator is intentionally pure and preview-only. It does not read the database, write the database, read files, call APIs, connect product import routes, or add UI behavior.

## Non-Goals

- Not a DB write implementation.
- Not a DB read implementation.
- Not a DB schema migration.
- Not a product import route connection.
- Not an API change.
- Not a UI import button or workflow.
- Not a raw HRC zip reader.
- Not an external artifact JSON reader.
- Not an import execution command.

## Inputs

The validator accepts only in-memory values:

- `rows`: `HrcImportPreviewRow[]`
- `existingCanonicalKeys`: `string[]`
- optional normalization settings

`existingCanonicalKeys` is supplied by the caller as a snapshot. The validator does not fetch it from the DB.

## Outputs

The validator returns:

- validated rows with `validationStatus`
- canonical key preview result
- import preview allowance
- DB write allowance, always `false`
- reasons and warnings
- summary counts

## Validation Statuses

- `READY_FOR_IMPORT_PREVIEW`
- `BLOCKED_BY_PREVIEW_DECISION`
- `MISSING_CANONICAL_KEY`
- `DUPLICATE_IN_BATCH`
- `DUPLICATE_EXISTING_DB`
- `PRIVACY_BLOCKED`
- `DB_WRITE_NOT_ALLOWED`
- `EXCLUDED`

## Duplicate Policy

Duplicate checks are preview-only:

- If a ready preview row has no `canonicalKeyPreview`, it is `MISSING_CANONICAL_KEY`.
- If the same canonical key appears more than once in the batch, ready preview rows with that key are `DUPLICATE_IN_BATCH`.
- If a ready preview row's canonical key appears in externally supplied `existingCanonicalKeys`, it is `DUPLICATE_EXISTING_DB`.
- The validator does not read existing keys from the product DB.
- The validator does not skip, update, insert, or mutate rows.

## Manual Review And Exclusion Policy

- `NEEDS_MANUAL_REVIEW` rows remain `BLOCKED_BY_PREVIEW_DECISION`.
- `HOLD` rows remain `BLOCKED_BY_PREVIEW_DECISION`.
- `EXCLUDE` rows are `EXCLUDED`.
- Privacy failures are `PRIVACY_BLOCKED`.
- These statuses keep `importPreviewAllowed` false.

## DB Write Policy

`dbWriteAllowed` is always `false`.

`assertNoHrcImportDbWriteAllowed` reports whether any source row tried to carry a true DB-write flag. Even when unsafe input is detected, validated output keeps `dbWriteAllowed` false.

## v2.9 Fixture Expectation

Using the v2.9 classification fixture:

- total: 28
- ready for import preview: 19
- blocked by preview decision: 8
- excluded: 1
- duplicate in batch: 0
- duplicate existing DB: 0
- missing canonical key: 0
- DB write allowed true count: 0

When one import candidate key is supplied through `existingCanonicalKeys`, `duplicateExistingDbCount` becomes 1.
