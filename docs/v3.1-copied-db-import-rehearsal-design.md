# v3.1 Copied DB Import Rehearsal Design

## Purpose

v3.1 evaluates how a real HRC-derived import rehearsal could be performed safely against a copied database only. The original production database is not a write target.

This design builds on the v3.0 no-write product import foundation. It documents a future rehearsal procedure, approval boundary, rollback policy, and evidence requirements before any implementation step can make a copied DB write possible.

## Core Principles

- The original production DB is never a write target.
- Only a copied DB can be considered as a rehearsal target.
- A copied DB target must live outside the normal repo working database path, under an approved restore-test or backup-root rehearsal area.
- Copied DB writes remain impossible unless a later step adds an explicit approval gate.
- Production DB targets are rejected even if an approval token or flag is present.
- Rehearsal results are evidence, not release artifacts, and must not be committed to the original repo by default.
- Raw HRC zip files remain repo-external and are not read, copied, extracted, or committed by this design step.
- Product import routes, API flows, UI flows, and package scripts remain disconnected.

## Step Design

### Step A: Copied DB Creation Procedure Design

Future implementation may copy the verified source archive DB or a separately approved production DB snapshot into a restore-test or backup-root rehearsal folder. This step only defines that policy; it does not create a copied DB.

Required design constraints:

- Source DB SHA must be recorded before copy.
- Target path must pass copied DB path guard checks.
- DB sidecar files must not be copied as rehearsal inputs.
- Copy output must remain outside the normal production DB location.
- Rehearsal folders must be excluded from commits and release source archives.

### Step B: Copied DB SHA Baseline

Before any future rehearsal write, the copied DB SHA must be recorded as the baseline. The original production DB SHA must also be recorded before and after the rehearsal and must remain unchanged.

Required evidence:

- `productionDbSha256Before`
- `productionDbSha256After`
- `copiedDbSha256Before`
- expected copied DB target path decision
- safety checks showing production DB is not targeted

### Step C: Import Preview Validation Reuse

The v3.0 import preview contract and validator remain the first gate. A copied DB rehearsal cannot proceed unless preview validation passes for the selected candidate set.

Required validation:

- import candidate classification is present
- canonical keys are available
- blocking privacy/path exposure is absent
- preview rows are deterministic
- write-capable state remains disabled until approval is evaluated

### Step D: Duplicate and Canonical Key Validation Reuse

The existing solution snapshot adapter and preview validator should be reused to detect conflicts before any copied DB write rehearsal.

Required checks:

- duplicate canonical keys in the candidate batch
- duplicate canonical keys against the existing solution snapshot
- missing canonical keys
- excluded candidates
- manual-review candidates that must not be promoted silently

### Step E: Copied DB Write Rehearsal Gate

A future copied DB write rehearsal must require an explicit approval token or approved flag. Approval applies only to a copied DB target that passed the path guard.

Gate rules:

- No approval means copied DB write is impossible.
- Approval is scoped to the copied DB path decision.
- Production DB target is rejected even with approval.
- Repo-local production DB target is rejected.
- Raw HRC zip path input remains out of scope.
- API/UI/package script exposure remains out of scope unless separately planned.

### Step F: Rollback and Restore Verification

Rollback verification must prove that the copied DB can return to its pre-rehearsal state.

Required rollback evidence:

- copied DB SHA before rehearsal
- copied DB SHA after rehearsal
- rollback source or pre-rehearsal backup reference
- copied DB SHA after rollback
- confirmation that production DB SHA before and after is unchanged
- confirmation that no production DB write occurred

### Step G: Post-Rehearsal Diff and Report Design

Future reporting should summarize rehearsal effects without committing generated reports by default.

Report design candidates:

- candidate counts
- inserted/updated/skipped/error counts against the copied DB
- duplicate and manual-review outcomes
- copied DB SHA before and after
- rollback verification result
- production DB SHA unchanged result
- privacy/path scan result

Generated rehearsal reports must remain outside committed repo history unless a later step separately approves sanitized documentation.

## Approval Gate Policy

The approval gate exists to prevent accidental write behavior.

- Approval must be explicit.
- Approval must be bound to a copied DB target.
- Approval must not apply to the production DB.
- Approval must not enable product import routes, API flows, UI flows, package scripts, raw zip reads, or report JSON writes by itself.
- Approval evidence must be recorded in the rehearsal manifest or equivalent summary.

Examples of future approval evidence:

- approval token value category, never a private token
- approved copied DB target decision
- branch and commit hash
- DB SHA baseline values
- preview validation pass result
- rollback plan available result

## Rollback Policy

Rollback is mandatory for any future copied DB write rehearsal.

- Record copied DB SHA before rehearsal.
- Record copied DB SHA after rehearsal.
- Restore the copied DB from the pre-rehearsal backup or verified copy.
- Record copied DB SHA after rollback.
- Confirm production DB SHA before and after remains unchanged.
- Treat rollback failure as a blocking rehearsal failure.

## Non-goals

- No production DB import implementation.
- No original production DB write.
- No DB schema migration.
- No product import API route connection.
- No UI import flow.
- No `package.json` import or preview script.
- No raw HRC zip direct import.
- No raw HRC zip access, copy, extraction, or commit.
- No public or GitHub release work.
- No generated artifact JSON commit.

## Step 2 Candidates

The safest next step is one of:

- Copied DB rehearsal guard helper.
- Copied DB target approval contract helper.

Either option should remain pure-helper-only unless separately approved. It should not create a copied DB, write a DB, connect an API/UI route, or add a package script.
