# v4.0 entry criteria

## Purpose

This document defines the gate for moving from v3.9 to v4.0.

v4.0 should begin only when v3.9 has completed practical stabilization and no
known blocker remains in the public-facing study workflows.

## Required Conditions

All of the following must be true before v4.0 starts:

- v3.9 Must items are complete or explicitly reclassified with evidence.
- `npm.cmd run typecheck` passes.
- `npm.cmd run test` passes.
- `npm.cmd run build` passes.
- `npm.cmd run test:smoke` passes.
- Analyze, Database, Trainer, and public-readiness flows have no known blocker.
- README matches the latest public release state.
- RELEASE_HISTORY matches the latest public release state.
- Korean-first visible copy and accessible-name baselines are maintained.
- localStorage migration and fallback behavior remains safe.
- public safety scan has no blocker.
- no bundled DB/raw/generated artifact is tracked.
- package and lockfile changes are intentional and approved.
- original repo remote remains absent during original-repo release work.
- public upload can proceed without force push or tag overwrite.
- v4.0 has a clear user value beyond version numbering.

## Blockers

Treat the following as blockers:

- failed typecheck/test/build/smoke
- stale current public tag/status text in README or RELEASE_HISTORY
- major user-flow confusion between Analyze, Database, and Trainer
- broken localStorage fallback for presets, recent analyses, Trainer history, or
  Trainer filters
- inaccessible core controls that block keyboard or screen-reader use
- narrow layout that hides core controls
- concrete data-origin disclosure in public-facing materials
- bundled data/raw/generated artifacts
- live/RTA/OCR/screen capture/overlay/hotkey/watcher/client integration
- unapproved dependency or lockfile change
- original repo remote unexpectedly present
- public upload requiring force push or tag overwrite

## Non-Blocking Notes

The following can remain non-blocking if documented and verified:

- GitHub Actions runtime deprecation annotation when the workflow still passes
- intentionally shorter sanitized public history
- technical constants such as `EV`, `BB`, `canonical key`, or status enum values
- historical planning docs that are clearly not current public release status

## v4.0 Start Recommendation

If v3.9 completes all Must items and verification remains green, v4.0 should
focus on a concrete release goal rather than another broad polish bundle.
