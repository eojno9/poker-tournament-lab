## Confirmed Raw HRC Node Shape

`settings.json` top-level keys:

- `handdata`
- `eqmodel`
- `treeconfig`
- `engine`

`nodes/0.json` top-level keys:

- `player`
- `street`
- `children`
- `sequence`
- `actions`
- `hands`

Node shape details:

- `actions`: array[3]
- each action has `type` and `amount`
- `hands`: 169 entries
- each hand has `weight`, `played`, and `evs`
- `played`: array[3]
- `evs`: array[3]
- `sequence`: array[6]
- each sequence item has `player`, `type`, `amount`, and `street`

## Difference From Current App Schema

### HRC Raw Shape

- Node-level `actions[]`
- Hand-level `hands[hand].played[]`
- Hand-level `hands[hand].evs[]`
- `played[]` and `evs[]` indexes map to the node-level `actions[]`
- Node path is represented by `sequence[]`

### Current App v2 Shape

- `hand -> actions[]`
- Each hand owns action entries directly
- Each action may carry action kind, size, frequency, EV, ChipEV, ICM EV, source label, and warnings

### Compatibility Gap

The raw HRC node shape appears structurally compatible with a future adapter, but it is not the same as current app v2 import shape. v2.4 should record this mismatch explicitly and avoid silently treating raw HRC node JSON as app-native v2 strategy JSON.

## Amount Semantics Policy

v2.4 does not infer whether raw HRC `actions[].amount` values are big blinds, chips, pot percentage, or another HRC-internal unit.

- Amount unit: `UNKNOWN`
- Amount interpretation: `RAW_HRC_AMOUNT_UNINTERPRETED`
- Size label policy: `PRESERVE_AS_RAW_SIZE_LABEL`
- BB conversion applied: `false`
- Chip conversion applied: `false`
- Raw amounts may be preserved in `rawSizeLabel` and source metadata candidate fields.
- Any bb/chip conversion should wait for HRC format documentation or multiple-sample verification.
- Product import routing remains disconnected during v2.4.

## Sanitize Principles

- Raw HRC zip remains outside the repo.
- Only a sanitized, minimal test fixture can be added.
- The fixture must be clearly marked as raw HRC shape sample data, not product-ready import payload.
- Preserve key-shape and compatibility-critical arrays:
  - settings key summary
  - node top-level keys
  - `actions[].type`
  - `actions[].amount`
  - 169 hand coverage when feasible
  - `hands[hand].weight`
  - `hands[hand].played[]`
  - `hands[hand].evs[]`
  - `sequence[]`
- Remove or avoid private paths, local directory names, player names, user names, nicknames, emails, and sensitive notes.
- Do not invent missing actions, sizes, frequencies, or EV values.
- Do not describe the fixture as a GTO recommendation.

## Fixture Design Direction

Recommended metadata:

- `sampleKind`: `REAL_HRC_RAW_EXPORT_SAMPLE`
- `sanitized`: `true`
- `originalTool`: `HRC`
- `rawZipCommitted`: `false`
- `originalFileName`: generalized or sanitized filename
- `streetScope`: `PREFLOP`
- `source`: `HRC_PRECOMPUTED_DB`
- `notes`: sanitized raw-shape compatibility sample, not product import data

Recommended fixture structure:

- `metadata`
- `settingsKeyShape`
- `nodeKeyShape`
- `rawShapeSummary`
- `settingsSample` or minimal sanitized settings subset
- `nodeSample` or minimal sanitized node subset
- `privacyScan`
- `compatibilityNotes`

If preserving all 169 hands is safe and small enough, keep 169 hand coverage. If not, preserve enough hands for shape tests and separately record observed full hand count from the raw candidate.

## Compatibility / Mismatch Report Direction

The report should include:

- raw candidate file name, sanitized/generalized
- raw zip committed: false
- settings keys
- node keys
- action count
- hand count
- sequence count
- action item key shape
- hand item key shape
- current app v2 validator pass/fail
- mismatch: raw node-level `actions[]` versus app `hand -> actions[]`
- mismatch: `played[]` and `evs[]` index mapping needs adapter
- amount semantics: raw HRC amounts are preserved without bb/chip inference
- source metadata candidate: raw node keys, settings keys, action types, action amounts, and conversion warnings
- spot candidate: node street/player and sequence are preserved without canonical key/import connection
- privacy scan result
- conclusion: adapter needed before product import

The report should be read-only and should not modify import/analyze/fallback/solver logic.
