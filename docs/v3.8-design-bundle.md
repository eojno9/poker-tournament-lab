This document fixes the v3.8 implementation scope before code changes. v3.8 is
not a new broad feature release. It is a v4.0 readiness gap-closure bundle for
Trainer stability, accessibility, state consistency, localStorage robustness,
Korean-first copy, and public readiness.

dependencies, DB/raw/generated artifacts, GitHub refs, or tags.

## Trainer State Transition Audit

### Current State

v3.7 exposes:

- start/not-started status
- in-progress status
- completed status once session attempts reach current candidate count
- reset action that clears current session counters and answer state
- mistake retry flow that returns the user to a practice spot

### v3.8 Target

Audit and harden these transitions:

| Transition | Expected result |
| --- | --- |
| no candidates -> filters changed to matching candidates | session returns to a usable not-started state |
| in-progress -> filters changed | current answer state and candidate index remain safe |
| in-progress -> candidate count reached | completion copy and summary appear once |
| completed -> reset | session counters reset without clearing all history |
| completed -> filter reset | completion state does not imply a stale candidate set |
| mistake retry -> answer -> resolve/dismiss | local queue state remains consistent |
| storage write failure during answer | UI remains usable and current session counters stay local |

Recommended implementation approach:

- keep state local to `App.tsx` unless a small pure helper reduces test burden
- avoid a new state machine library
- prefer explicit derived-state tests over broad snapshots

## localStorage Key And Migration Audit

### Key Registry

v3.8 should document and verify the current Trainer localStorage keys:

| Key | Purpose | Current version |
| --- | --- | --- |
| `ptl.trainer.recent.v1` | recent local attempts | v1 |
| `ptl.trainer.mistakes.v1` | local mistake queue | v1 |
| `ptl.trainer.filters.v1` | local Trainer filter settings | v1 |

### Migration/Fallback Rules

Every Trainer storage path should:

- tolerate missing storage
- tolerate invalid JSON
- tolerate unknown versions
- tolerate partial payloads where a merge is safe
- tolerate write failure
- avoid throwing during UI initialization
- avoid storing source/provider/permission/private path details
- keep current in-memory session usable after persistence failure

### v3.8 Minimum

- Add or update tests that cover the three Trainer storage keys as a set.
- Ensure filter fallback behavior is documented in test names.
- Confirm recent/mistake history fallback remains compatible with existing data.
- Avoid introducing a new key unless the implementation makes a clear case.

## Accessibility And Keyboard Baseline

v3.8 should check core Trainer controls:

| Surface | Expectation |
| --- | --- |
| Main tab | Trainer tab is reachable and named |
| Filter controls | labels or accessible names identify each input |
| Save/load/reset filter buttons | keyboard reachable and clearly named |
| Session reset | visually distinct and clearly named |
| Answer buttons | keyboard reachable with Korean-first action names |
| Mistake filters | selected state is understandable |
| Retry/hide/resolve controls | keyboard reachable and scoped to the row |
| Clear-history actions | destructive intent is clear |

This is a baseline check, not a full accessibility certification.

Implementation should prefer semantic HTML and existing button/select patterns.
No dependency should be added.

## Responsive And Narrow Layout Baseline

Check at least one narrow viewport for:

- Trainer top guidance and safety copy
- filter controls
- session status card
- problem card
- answer buttons
- feedback/summary block
- mistake filter controls
- mistake rows
- local summary/statistics blocks

Expected result:

- no text overlap
- buttons remain readable
- counters do not force horizontal scrolling
- empty states remain visible
- Korean text wraps cleanly

If automation is expensive, document manual QA notes and add smoke selectors
where practical.

## Korean-First Copy Audit

Audit visible user-facing Trainer copy:

| Surface | Criteria |
| --- | --- |
| Trainer title/guidance | Korean-first and off-table |
| Session status | Korean labels for start/progress/complete/reset |
| Filters | Korean labels and clear storage copy |
| Mistake review | Korean labels for all/unresolved/resolved/hidden |
| Empty states | Korean, actionable, calm |
| Errors/fallbacks | Korean where shown to the user |
| Summary/statistics | Korean labels for session/recent/local scope |
| Safety copy | no live/RTA/client ambiguity |

Technical/domain labels such as `EV`, `BB`, and `canonical key` may remain when
needed for inspection.

## Error, Empty, And Loading State Consistency

v3.8 should review the Trainer user experience for:

- no Trainer candidates
- filtered mistake list is empty
- no local history yet
- localStorage unavailable
- localStorage corrupt payload
- save/load/reset success and failure notices
- completed session with no valid candidates after filter changes

Copy should avoid implementation jargon and should not suggest live assistance
or remote persistence.
