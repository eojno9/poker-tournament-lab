# poker-tournament-lab v2.6 Plan

## v2.6 Summary

v2.6 plans a read-only dry-run artifact reporting track for real HRC raw zip compatibility checks.

v2.5 proved that a repo-external raw HRC zip can be inspected without extraction, product import route connection, DB writes, API changes, or UI changes. It fixed the dry-run report schema, status enum, malformed/privacy warning behavior, multiple node selection policy, `adapterReportSummary`, `validatorResult`, and `mismatchSummary`.

v2.6 does not connect raw HRC exports to production import routes. The goal is to design an artifact-safe JSON report shape so dry-run results can later be saved under `artifacts/hrc-dry-run-reports/` and compared across multiple raw HRC zip candidates.

## Dry-Run Artifact Report Design

Future v2.6 implementation steps should convert `buildHrcRawZipDryRunReport` output into an artifact-safe JSON document.

The planned artifact location is:

- `artifacts/hrc-dry-run-reports/`

This location is a design target only in the plan. The planning step must not create the directory or any report files.

Artifact reports must:

- store no raw HRC zip files
- store no full raw zip extraction output
- store no raw path in clear text
- store no long hand strategy payload dump
- store shape, validator, mismatch, privacy, amount, and verification summaries only
- keep mismatch samples capped at 3 entries
- keep privacy warning details category-based and redacted
- keep `productImportConnected: false`
- keep `dbWriteApplied: false`
- keep `apiUsed: false`
- keep `uiUsed: false`

## Artifact File Naming Policy

Artifact file names should be deterministic enough for sorting and safe enough for sharing.

Planned file pattern:

`artifacts/hrc-dry-run-reports/hrc-dry-run-YYYYMMDD-HHMMSS-<sanitized-name>.json`

Rules:

- Use local timestamp at report generation time.
- Include only a sanitized file-name hint, never a full local path.
- Strip path separators, drive letters, account names, user directories, and email-like strings.
- Keep the sanitized name short enough for Windows path safety.
- If the source name cannot be safely sanitized, use `unknown-source`.
- Do not include raw zip contents or extracted file names beyond safe entry summaries already present in the report.

## Privacy And Safety Policy

- Raw HRC zip originals remain outside the repo.
- Raw HRC zips must not be committed.
- Full raw zip extraction output must not be committed.
- Artifact reports must not store raw local paths.
- Artifact reports must not store raw hand strategy payload dumps.
- Artifact reports must not store privacy-sensitive values.
- Privacy warning matches must be redacted to categories.
- Mismatch sample size is capped at 3.
- Privacy warning status must not promote any product import candidate.
- Artifact output must not imply GTO recommendation quality.
- Artifact output must not imply product import readiness.

Sensitive patterns remain:

- `<local-user-path>`
- `AppData`
- `Desktop`
- `Documents`
- account/user names
- player names
- nicknames
- screen names
- email addresses
- sensitive notes

## Amount Semantics Policy

v2.6 keeps the v2.5 amount policy unchanged:

- Do not infer whether HRC `amount` is bb, chip, pot fraction, or an internal unit.
- Keep `amountUnit` as `UNKNOWN`.
- Keep `amountInterpretation` as `RAW_HRC_AMOUNT_UNINTERPRETED`.
- Preserve raw amount values only as raw labels or source metadata candidate fields.
- Keep `bbConversionApplied` as `false`.
- Keep `chipConversionApplied` as `false`.
- Do not add bb/chip conversion until HRC format documentation or multiple raw samples confirm the semantics.

## Multiple Node Policy

v2.6 keeps the v2.5 node policy unchanged:

- Prefer `nodes/0.json` when it exists.
- If `nodes/0.json` is absent, select the first `nodes/*.json` entry by deterministic string sort.
- Record the decision in `selectedNodeEntry` and `selectedNodeReason`.
- Record the total node count in `nodeEntryCount`.
- Record a short `nodeEntriesSample`.
- Add `MULTIPLE_NODE_ENTRIES` to warnings when multiple nodes are present.
- Keep `multiNodeAggregationApplied: false`.
- Do not merge multiple nodes into one product candidate.
- Treat multi-node aggregation as future work after product import schema and node semantics are explicitly approved.
