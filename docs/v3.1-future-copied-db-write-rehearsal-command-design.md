# v3.1 Future Copied DB Write Rehearsal Command Design

## Purpose

This docs-only design defines the safety boundary for a possible future copied DB write rehearsal command. It builds on the Step 2 through Step 6 helper chain:

1. copied DB target approval contract
2. copied DB rehearsal guard
3. copied DB rehearsal plan builder
4. copied DB rehearsal report shape
5. copied DB rehearsal dry-run orchestrator

The command would use the helper chain to classify whether a copied DB rehearsal is ready, blocked, or dry-run-only. This step does not implement the command.

## Command Non-goals

A future copied DB write rehearsal command is not:

- a production DB import command
- a repo-local DB write command
- a raw HRC zip import command
- an API/UI product import flow
- an automatic `package.json` script
- a default report JSON file writer
- a DB schema migration command
- a rollback executor without separate approval

The v3.1 baseline production DB SHA remains:

`92BBF8DA75EA34ABF4CBCF8111ADBCD729DA93D7DDE690E14EB52482A8E16461`

## Candidate Command Names

Future command names may include:

- `import:hrc:copied-db-rehearsal:dry-run`
- `import:hrc:copied-db-rehearsal:write`

These names are examples only. This step does not add a `package.json` script and does not create a command entrypoint.

## Required Inputs

A future command must require enough precomputed evidence to run the pure helper chain safely:

- copied DB target path, reported only through a redacted summary
- `targetKind: COPIED_DB`
- `targetLocationKind: RESTORE_TEST` or `BACKUP_ROOT_REHEARSAL`
- `originalDbShaBefore`
- `originalDbShaAfter`
- `copiedDbShaBefore`
- preview validation result
- duplicate/canonical key validation result
- rollback plan evidence
- privacy scan result
- raw zip absence check
- artifact report absence check
- product import route disconnected evidence
- API/UI runtime unchanged evidence

The command must never print or persist an unredacted local path, email, private token, raw zip path, or raw HRC source folder.

## Approval Gate

Dry-run mode is allowed only while write flags remain false:

- `writeAllowed: false`
- `productionDbWriteAllowed: false`
- `reportFileWriteAllowed: false`

Copied DB write rehearsal mode requires explicit approval evidence, such as a dedicated flag or token, and that approval applies only to a copied DB target. A production DB target is rejected even with approval. A repo-local DB target is also rejected.

Approval must not imply production DB write access, schema migration access, product import route activation, package script exposure, or report JSON file writing.

## Exit Code Policy

A future command should use the same exit code policy as the report/orchestrator helpers:

- `0`: ready or success shape
- `1`: validation or plan blocked
- `2`: safety blocked
- `3`: privacy/path blocked
- `4`: input or internal shape blocked

Manual review candidates may be non-fatal only if validation and safety gates remain passing. Duplicate, SHA, privacy, target, raw zip, or execution evidence failures are fatal.

## Output and Report Policy

Default output should be a console summary generated from safe, redacted helper output.

Report JSON file writing remains disabled by default:

- `reportFileWriteAllowed: false`
- `canWriteReportFile: false`

Introducing report file writing requires a separate planning and approval step. The command output must not expose real local paths, email addresses, private tokens, raw HRC zip paths, or raw HRC source folders.

## Future Implementation Checklist

Before any command entrypoint is implemented:

- confirm the command remains copied-DB-only
- confirm no production DB write path exists
- confirm no repo-local DB write target is accepted
- confirm no `package.json` script is added without separate approval
- confirm no product import API/UI route is connected
- confirm no raw HRC zip access is introduced
- confirm no report JSON file is written by default
- test approval flag/token behavior
- test production DB and repo-local target rejection
- test exit code mapping
- test output redaction

Before any `package.json` script is added:

- document the script name and exact command
- prove the command defaults to dry-run or copied-DB-only scope
- keep production DB writes impossible
- keep report file writing disabled unless separately approved

Before any copied DB write rehearsal:

- verify copied DB target guard output
- record original DB SHA before and after
- record copied DB SHA before rehearsal
- require rollback plan evidence
- confirm privacy scan pass
- confirm raw zip absence
- confirm artifact report absence
- confirm product import route remains disconnected

Before any rollback rehearsal:

- keep the rollback action inside a restore-test or backup-root rehearsal area
- record copied DB SHA before rollback
- record copied DB SHA after rollback
- confirm original DB SHA remains unchanged

Restore-test cleanup must be explicit and scoped to the restore-test area. Broad repo-root cleanup remains forbidden.
