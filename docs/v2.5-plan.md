## Raw HRC Zip Dry-Run Import Adapter Concept

The v2.5 adapter track should remain a pure read-only pathway:

1. Accept a repo-external raw HRC zip path as input.
2. Confirm the path is outside the repo.
3. List zip entries.
4. Read `settings.json` in memory.
5. Read selected `nodes/*.json` entries in memory.
6. Recognize raw HRC node shape.
7. Apply `convertHrcRawNodeToMultiActionStrategy`.
8. Build source metadata candidate.
9. Build spot candidate.
10. Build validator candidate.
11. Compare candidate with current app v2 validator expectations.
12. Emit a dry-run report.
13. Perform no DB write.

This flow is not a product import endpoint. It is a compatibility and adapter report path.

## Dry-Run Pipeline

Recommended stages:

- Raw zip path input:
  - absolute path only
  - path must not be inside repo
  - path should be masked in reports
- Zip entry list:
  - entry count
  - top-level files
  - `settings.json` presence
  - `nodes/*.json` count
  - malformed or unsupported entries
- Settings read:
  - read in memory only
  - report top-level keys
  - scan for private path/email/player-name patterns
- Node read:
  - read node JSON in memory only
  - detect `actions[]`, `hands`, `played[]`, `evs[]`, and `sequence[]`
  - report malformed nodes without changing product logic
- Raw node recognition:
  - action count
  - hand count
  - sequence length
  - sampled hand shape
  - length mismatches
- Adapter convert:
  - node `actions[index]`
  - hand `played[index]` -> frequency
  - hand `evs[index]` -> EV
  - unknown action warnings
- Source metadata candidate:
  - original tool
  - source shape
  - target shape
  - raw action types
  - raw action amounts
  - amount semantics
  - raw zip committed: false
- Spot candidate:
  - node player
  - street
  - sequence
  - action path candidate
  - decision node candidate
  - UNKNOWN/null for unavailable values
- Validator candidate:
  - app v2 hand -> `actions[]` candidate
  - validator pass/fail
  - missing required fields
  - mismatch list
- Report:
  - safe summary only
  - masked raw path
  - no raw payload dump
  - no DB write

## Node Selection Policy

v2.5 dry-run reports inspect one raw HRC node at a time.

- Prefer `nodes/0.json` when it exists.
- If `nodes/0.json` is absent, select the first `nodes/*.json` entry by deterministic string sort.
- Record the decision in `selectedNodeEntry` and `selectedNodeReason`.
- Record the total node count in `nodeEntryCount`.
- Record only a short `nodeEntriesSample` to avoid noisy reports.
- When multiple nodes are present, add `MULTIPLE_NODE_ENTRIES` to warnings.
- Keep `multiNodeAggregationApplied: false`.
- Do not merge multiple nodes into one product candidate in v2.5.
- Treat multi-node aggregation as future work after product import schema and node semantics are explicitly approved.

## Amount Semantics Policy

v2.5 must keep the v2.4 amount policy unchanged:

- Do not infer whether HRC `amount` is bb, chip, pot fraction, or an internal unit.
- Keep `amountUnit` as `UNKNOWN`.
- Keep `amountInterpretation` as `RAW_HRC_AMOUNT_UNINTERPRETED`.
- Preserve raw amount values only as `rawSizeLabel` or source metadata candidate fields.
- Keep `bbConversionApplied` as `false`.
- Keep `chipConversionApplied` as `false`.
- Do not add bb/chip conversion until HRC format documentation or multiple raw samples confirm the semantics.

## Privacy And Sanitize Policy

- Raw HRC zip originals remain outside the repo.
- Raw HRC zips must not be committed.
- Full raw zip extraction output must not be committed.
- Dry-run helpers may read entries in memory or use repo-external temporary locations only when strictly needed.
- Reports must mask raw paths.
- Reports must not dump full raw payload contents.
- Scan for:
  - `C:\Users\`
  - `AppData`
  - `Desktop`
  - `Documents`
  - account/user names
  - player names
  - nicknames
  - screen names
  - email addresses
  - sensitive notes
- Stop and report if privacy-sensitive data cannot be safely masked.
