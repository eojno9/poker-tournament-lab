# v3.9 design bundle

## Design Intent

v3.9 is a final stabilization bundle. It should make the current app easier to
understand, safer to review, and less surprising in edge states before v4.0.

The bundle avoids new persistence, new dependencies, and broad UI redesign.

## User Flow Design

### Analyze

Design goals:

- Keep current form, validation, preset, and recent-analysis behavior.
- Improve Korean-first labels and accessible names where older text remains
  English-only.
- Make invalid input and request failure states explain what the user can do
  next.
- Keep fallback and not-solved states clearly labeled as non-exact or missing
  exact results.

Implementation notes:

- Do not change request payload shape.
- Do not add auto-analysis or hidden background calls.
- Preserve localStorage-only presets/recent analyses.

### Database

Design goals:

- Keep search/detail/filter behavior.
- Make the Database to Analyze handoff explicit after the user fills Analyze
  from a database result.
- Preserve existing reset paths.
- Avoid implying that a database result is bundled in public source.

Implementation notes:

- Add copy or status UI rather than a new cross-screen state model.
- Keep canonical key and technical labels where needed.

### Trainer

Design goals:

- Preserve v3.8 session and storage safeguards.
- Only adjust Trainer copy/layout if needed for consistency with the other
  screens.
- Do not add new Trainer persistence or server sync.

Implementation notes:

- Treat Trainer as mostly stable after v3.8.
- Any Trainer changes should be narrow and covered by existing smoke tests.

### Public Readiness

Design goals:

- Keep original README and RELEASE_HISTORY aligned with the latest public
  release before future upload work.
- Keep SECURITY guidance public-safe and free of sensitive details.
- Keep CONTRIBUTING aligned with local-only data and safety rules.

Implementation notes:

- Add status checks that catch stale current-tag wording.
- Do not document private development details publicly.

## State And Storage Design

Storage remains browser-local only.

Current key families:

- Analyze presets
- Recent analyses
- Trainer recent attempts
- Trainer mistakes
- Trainer filters

v3.9 should not introduce a new storage version unless a concrete migration need
is found. If a helper or inventory is added, it should be minimal and should not
change existing payload shape.

## Korean-First Copy

Rules:

- visible labels, buttons, empty states, and recovery guidance should be Korean
  first
- accessible names should be Korean-first where they are user-facing
- technical constants may remain as exact values
- English-only strings in older screens should be reviewed before v4.0

Examples of acceptable technical labels:

- `EV`
- `BB`
- `canonical key`
- `HRC_PRECOMPUTED_DB`
- `NOT_SOLVED`

## Accessibility And Responsive Design

v3.9 should preserve v3.8 Trainer improvements and extend the audit to older
screens:

- focus order is predictable
- controls have accessible names
- selected or active states do not rely on color alone
- disabled states are visually and behaviorally clear
- long technical labels wrap safely
- narrow layouts do not hide core controls

## Public-Safe Impact

The implementation must not add:

- bundled data artifacts
- concrete data-origin details
- private local paths or account names
- secrets or token examples
- live/RTA/OCR/screen capture/overlay/hotkey/watcher/client integration

## Implementation Order

1. Sync README/RELEASE_HISTORY status wording.
2. Review and patch Korean-first accessible names/copy on Analyze, Database,
   Browser, and artifact surfaces.
3. Clarify Database to Analyze handoff and reset/status copy.
4. Normalize empty/error/not-solved/storage copy where touched.
5. Add focused tests/smoke assertions.
6. Optionally triage CI action deprecation warning if low-risk.
7. Run full verification.
