# v4.0 release acceptance criteria

## Purpose

This document defines the minimum criteria for a v4.0 public release.

## Required Verification

All release candidates must pass:

```powershell
npm.cmd run typecheck
npm.cmd run test
npm.cmd run build
npm.cmd run test:smoke
```

Expected current test baseline:

- core: `383`
- server: `82`
- web: `119`
- smoke: `7/7`

Counts may change only if tests are intentionally added or removed and the
change is documented.

## Functional Acceptance

- Entry/navigation flow is understandable.
- Database and Browser review flows remain read-only for public source.
- Database to Analyze handoff is explicit, editable, and does not auto-run
  analysis.
- Analyze validation, presets, recent analyses, exact/fallback/not-solved
  states remain stable.
- Trainer session, filters, mistakes, retry/resolved/dismissed states, summary,
  and local statistics remain stable.
- Verification and canonical-key copy remain understandable.

## State Acceptance

- idle, ready, loading/processing, success, empty, invalid, not solved,
  unavailable, recoverable error, and completed states use consistent
  Korean-first copy where user-facing.
- Reset scopes are explicit and do not delete unrelated local data.
- Duplicate or rapid actions do not create known state corruption.

## Local Data Acceptance

- Analyze preset and recent-analysis storage fallback remains safe.
- Trainer recent, mistake, and filter storage fallback remains safe.
- Legacy Trainer recent migration remains compatible.
- No server-side user history, account sync, or cloud persistence is added.

## Public Readiness Acceptance

- README, SECURITY, CONTRIBUTING, and RELEASE_HISTORY are consistent.
- SECURITY.md reporting guidance is readable and public-safe.
- Public wording does not disclose data source/provider/permission details.
- No secrets, private paths, private emails, DB files, raw archives, generated
  reports, `node_modules`, `dist`, `test-results`, or artifacts are tracked.
- Sanitized public upload can proceed without force push or tag overwrite.

## CI Acceptance

- GitHub Actions succeeds after public upload.
- Node SQLite experimental warning remains non-blocking.
- GitHub Actions Node runtime deprecation annotation is acceptable only if
  documented as non-blocking or addressed with verified official action updates.

Implementation-stage CI triage:

- Official remote tags were checked for `actions/checkout` and
  `actions/setup-node`.
- Both actions expose stable `v7` tags.
- The workflow was updated to `actions/checkout@v7` and `actions/setup-node@v7`
  only.
- Node.js remains `24`.
- No third-party action, package dependency, package-lock change, or workflow
  shape change was introduced.

## Release Mechanics

- candidate tag: `v4.0`
- candidate title: `v4.0 - Stable off-table study milestone`
- original repo must keep remote absent
- public upload must use a fresh sanitized snapshot
- normal push only
- no `git push --tags`
- no tag move, delete, or overwrite
- status hotfix after tagging may update `main` only
