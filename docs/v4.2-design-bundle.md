# v4.2 design bundle

## Design intent

v4.2 is a maintenance-only bundle. It narrows component ownership, closes one
server error-disclosure boundary, makes release validation reusable in a
public clone, and improves regression diagnostics without changing the study
workflow.

## HRC artifact component boundary

Create a focused `HrcArtifactsView` module that owns:

- list, loading, and list-error state
- filters and derived filtered items
- selected filename
- detail, loading, and detail-error state
- initial and manual list refresh
- GET-only list/detail API calls
- list, invalid-item, detail, and JSON-preview rendering helpers

`App.tsx` owns only tab selection and renders `<HrcArtifactsView />`. The new
module must not import from `App.tsx` or receive a large prop surface.

If the existing generic detail row prevents extraction, move only that small
presentational primitive to a neutral module or keep a local equivalent with
the same DOM contract. Do not introduce a component framework.

## State and callback ownership

| State/callback | Owner |
| --- | --- |
| active tab | `App` shell |
| HRC list/detail/filter state | `HrcArtifactsView` |
| artifact sanitization and summary/filter algorithms | existing `hrcArtifactDashboard` helper |
| GET request implementation and response types | existing `api` module |
| Database-to-Analyze prefill state | `App` shell until a later boundary |
| Analyze form, presets, recent history, and execution | `AnalyzeView` |

The extraction must preserve request timing and cancellation behavior. No new
write/import/batch/raw-archive action is permitted.

## Analyze handoff boundary

Do not extract the full Analyze view in v4.2. Its form, JSON mode, presets,
recent history, action-sizing candidates, result state, and handoff notices
form a broad state surface.

A helper-only Should item may move:

- `AnalyzeHandoffContext`
- `AnalyzePrefillPayload`
- validation of a handoff spot
- construction of Database handoff context
- pure conversion of a spot to Analyze form state

The App shell still owns the transient prefill and tab transition. Automatic
analysis remains prohibited; the user must explicitly run Analyze.

## Server error contract

Keep successful response shapes unchanged. For errors, use an additive stable
identifier rather than a breaking envelope migration.

Candidate shape:

```ts
interface SafeApiErrorBody {
  error: string;
  code: "INVALID_REQUEST" | "NOT_FOUND" | "UNPROCESSABLE" | "INTERNAL_ERROR";
}
```

Rules:

- explicit validation paths may keep their existing public-safe `error` text
  and add a code
- the global handler returns status 500, `INTERNAL_ERROR`, and a generic
  message
- exception messages, stack traces, payloads, local paths, and DB details are
  not returned by the global handler
- the browser continues mapping HTTP status to Korean-first user copy and does
  not render response bodies
- no logging framework or external telemetry is added

## Smoke helper boundary

Keep scenario bodies readable. Good helper boundaries are:

- `installDefaultApiMocks(page)`
- `fulfillJson(route, body, status?)`
- `openTab(page, accessibleName)`
- `expectNoDocumentHorizontalOverflow(page)`
- fixture constants in one public-safe module when moving them clearly reduces
  noise

Do not hide core assertions in a generic DSL. The seven user-intent scenarios
remain visible and independently runnable.

## Viewport test plan

Use existing Playwright and CSS; add no visual-test dependency. At one narrow
viewport, verify:

- document width does not exceed viewport width beyond a one-pixel tolerance
- HRC filters, list/detail container, and refresh/detail controls remain
  reachable
- one dense Analyze or Database state keeps its primary action visible
- long technical labels wrap or scroll inside their owned region rather than
  expanding the document

This is a structural responsive baseline, not pixel-perfect visual regression.

## Release preflight modes

Proposed CLI:

```powershell
node scripts/public-release-preflight.mjs --mode original `
  --candidate-tag v4.2 --expected-current-public-tag v4.1

node scripts/public-release-preflight.mjs --mode public `
  --candidate-tag v4.2 --expected-current-public-tag v4.1 `
  --expected-origin https://github.com/eojno9/poker-tournament-lab.git
```

Mode rules:

- `original`: clean tree, no remote, candidate local tag absent, public text
  still names the current completed tag
- `public`: clean tree, approved origin present, branch/ref relationship
  readable, candidate remote tag absent, package/artifact/public-text gates
  pass
- both modes only read files and invoke read-only Git commands
- no mode creates/deletes files, changes remotes, creates tags, commits, or
  pushes
- optional `--format json` writes only to stdout; no report file is generated

Actions remains a post-push external gate and is not queried or mutated by the
preflight script.

## Status hotfix policy

Before the snapshot commit, README and RELEASE_HISTORY name the candidate as
the current public tag only when the same reviewed snapshot is about to receive
that tag. Durable text does not claim an Actions result that has not happened.

After push, a main-only docs hotfix is allowed only for an actual rendered
status error. It never moves or re-pushes the release tag. Exact Actions run IDs
are verification records, not mandatory current-status text.

## Safety and regression invariants

- Korean-first user messages remain the default
- off-table and no-live-integration boundaries remain visible
- storage keys, versions, migrations, and reset scopes remain unchanged
- no DB/raw/generated artifact is added
- no dependency or package-lock change
- no original repository remote
- no automatic release mutation
