# v4.1 plan

## Goal

v4.1 is a maintenance release for the stable v4.0 off-table study product. It
should reduce structural risk, make local persistence failure-safe, normalize
public error copy, and reduce repeated public-release status work without
changing the product scope.

## Non-Goals

v4.1 must not add:

- a solver or strategy-generation engine
- bundled DB, raw data, archives, or generated reports
- server-side user history, accounts, login, or cloud synchronization
- RTA, live assistance, OCR, screen capture, overlays, hotkeys, watchers, or
  poker-client integration
- a full UI redesign or broad App rewrite
- a storage framework or broad key/version migration
- a new framework or unapproved dependency/package-lock change
- automatic push, force push, tag creation, tag move, tag deletion, or tag
  overwrite
- an original-repository remote

## Must

1. Incremental component boundary
   - Extract `TrainerView` and its view-only formatting/table helpers from
     `App.tsx`.
   - Keep Trainer data loading, state, callbacks, and localStorage behavior
     inside the Trainer boundary.
   - Preserve current DOM semantics, test IDs, Korean-first copy, and safety
     copy.

2. Safe storage boundary
   - Introduce one small shared `StorageLike` and safe read/write/reset helper.
   - Catch storage access failures, including throwing `getItem`.
   - Keep domain parsing and shape validation in Analyze/Trainer modules.
   - Preserve all existing keys, versions, migration, limits, and reset scope.

3. Public-safe API error mapping
   - Stop using raw HTTP response bodies as default UI error messages.
   - Map network, validation, not-found, and server failures to concise
     Korean-first recovery copy.
   - Keep detailed internal payloads out of rendered text and browser logs by
     default.

4. Durable release status and read-only preflight
   - Replace the original README's stale v3.9 current-tag wording with accurate
     v4.0 status or a durable neutral link to public Git tags.
   - Keep RELEASE_HISTORY and README current-status sections consistent.
   - Add a no-write script that compares an expected release tag with README
     and RELEASE_HISTORY status text.
   - Fail with a non-zero exit on mismatch.
   - Keep tag creation, remote checks, scan review, commit, and push as explicit
     audited steps.

## Should

- Rename the stale `v1.2 smoke` suite to a stable workflow name.
- Extract reusable Playwright route fixtures or scenario setup without
  reducing the seven existing scenarios.
- Extract one additional low-coupling view only after the Trainer extraction is
  verified; HRC artifacts is the preferred candidate.
- Document a public docs subset/allowlist policy in a machine-checkable but
  read-only format.

## Could

- Add a small shared status notice component if active call sites have the same
  semantics and accessibility needs.
- Add a focused narrow-viewport assertion for an extracted view.
- Add a console-only summary to the release preflight script.


Focused tests:

- Trainer behavior and selectors remain unchanged after extraction.
- storage `getItem`, `setItem`, and reset failures return documented fallbacks.
- corrupt, partial, unknown, and legacy payload behavior remains compatible.
- API response bodies are not rendered as raw UI errors.
- HTTP/network error classes map to Korean-first next-action copy.
- release status preflight passes matching fixtures and fails stale fixtures.
- smoke scenario count and critical workflow assertions are not reduced.

## Implementation Result

The v4.1 maintenance implementation is complete locally:

- `TrainerView` now owns the existing Trainer loading, session, filter,
  grading, mistake-review, local-statistics, and fallback behavior.
- `App.tsx` delegates the Trainer tab to that boundary without changing its
  user flow, selectors, Korean-first copy, or safety scope.
- `safeStorage.ts` centralizes availability checks and exception-safe
  `getItem`, `setItem`, and `removeItem` operations. Domain parsing,
  validation, migration, record limits, and reset semantics remain in their
  original modules.
- Analyze presets, recent analyses, Trainer recent/current legacy history,
  mistakes, and filters retain their existing keys and versions.
- API requests no longer use raw response bodies as UI error messages.
  Network, request, not-found, unavailable, delayed, invalid-response, and
  server failures use Korean-first recovery copy.
- `public-release-preflight.mjs` performs read-only repository, package,
  artifact, public-text, release-status, and security-policy checks. It does
  not create tags, commits, remotes, clones, reports, or pushes.
- The smoke suite is now named `public workflow smoke`; all seven scenarios
  remain.

## Verification Result

- typecheck: PASS
- tests: PASS, core `383`, server `82`, web `130`
- build: PASS
- smoke: PASS, `7/7`
- preflight fixtures: PASS, one positive and ten negative cases

The first smoke run exposed two assertions that expected raw HTTP response
text. The UI correctly returned mapped Korean messages; those assertions were
updated to require the mapped text and reject the raw body. The rerun passed
all seven scenarios.

- A new narrow-viewport assertion was not added because this bundle made no
