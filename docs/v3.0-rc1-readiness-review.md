## v3.0-rc2 Follow-up Note

The v3.0-rc1 restore rehearsal later exposed release-process issues that were not product runtime issues:

- Strict source archive privacy scan found real local user/path tokens in committed source, tests, and documentation.
- Bundle clone DB SHA verification needed clearer policy because git bundles contain Git-tracked files only and may not include the untracked runtime SQLite DB.

The rc2 policy response is documented in `docs/v3.0-rc2-privacy-backup-fix.md`:

- Replace real local user/path tokens with neutral fixtures.
- Treat source archive restore as the file, privacy, current DB presence, and DB SHA verification path.
- Treat git bundle clone as the Git-tracked source, tag, checkout, and test/build/smoke verification path.
- Require any DB-dependent bundle clone verification to use explicit DB injection inside a restore-test folder only.
- Keep cleanup as a separate verification step and avoid repo-root destructive cleanup or PowerShell `-Include` based repo-wide cleanup.

This follow-up preserves the v3.0-rc1 tag and prepares a separate v3.0-rc2 candidate commit.
