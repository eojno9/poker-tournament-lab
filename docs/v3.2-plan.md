# v3.2 Copied DB Command Planning

## Summary

v3.2 starts from the completed v3.1 stable baseline and plans the next layer above the copied DB rehearsal foundation: command and entrypoint readiness. This step is docs-only. It does not implement an entrypoint, add a package script, write any database, connect API or UI flows, or push to GitHub.

## Goals

- Build on the v3.1 copied DB rehearsal foundation to safely plan or prepare the command and entrypoint layer.
- Review a copied DB command entrypoint implementation path without enabling production DB writes.
- Define an explicit approval gate before any package script can be added.
- Separately evaluate whether report JSON write should ever be introduced.
- Make the restore-test based copied DB rehearsal procedure more concrete before any real copied DB write rehearsal.
- Preserve the approval to guard to plan to report to orchestrator to output renderer flow from v3.1.
- Keep command output aligned with the v3.1 safe redacted renderer policy.

## Non-goals

- No production DB write.
- No repo-local DB write.
- No actual product import route connection.
- No API import workflow.
- No UI import workflow.
- No raw HRC zip direct processing.
- No GitHub push.
- No immediate `package.json` import or preview script.
- No immediate report JSON write.
- No production DB schema migration.
- No raw HRC zip copy, extraction, or commit.

## Safety Boundaries

- Original repo remote remains absent.
- GitHub push remains forbidden unless a later step explicitly approves it.
- Actual DB write remains forbidden, including copied DB write, until a later step explicitly approves the exact target and procedure.
- Any command entrypoint must remain disconnected from `package.json` scripts until a separate approved step.
- Any future output must use the v3.1 Step 9 renderer policy and must not expose real local paths, email addresses, private tokens, raw HRC paths, or raw zip paths.
- Copied DB rehearsal remains forbidden outside an approved restore-test or backup-root rehearsal area.
- Production DB targets are rejected even with approval.
- Repo-local DB targets are rejected even with approval.
- Report file write remains disabled until separately planned and approved.
- Product import API/UI surfaces remain disconnected.
