# v4.0 scope confirmation

## Purpose

v4.0 is justified only if it marks a stable public milestone rather than a
simple version bump. This document confirms whether the current product has a
clear user-facing value boundary after v3.9 and defines what must be finished
before the v4.0 public release.

## Proposed v4.0 Definition

v4.0 is the first stable public milestone where Analyze, Database, Trainer, and
verification/documentation surfaces are treated as one coherent off-table study
workflow. The milestone should make context handoff, state handling, local data
contracts, accessibility, release verification, and public distribution rules
stable enough to maintain after publication.

## Major-Version Justification

v4.0 is justified because v3.9 already completed the practical feature line:

- Analyze has form, validation, preset, recent-analysis, result, fallback, and
  not-solved behavior.
- Database and Browser support source review and Database to Analyze handoff.
- Trainer has local session, filter, mistake review, retry, summary, and local
  statistics behavior.
- Verification has core/server/web/smoke coverage and green public CI.
- Public release workflow has a working sanitized snapshot and tag process.

The remaining v4.0 value is not a broad new feature. It is stable-product
closure: polish the connected workflow, remove obvious public-quality issues,
lock the state/local-data contract, and define release acceptance criteria that
future maintenance can reuse.

## v3.9 Delta

v3.9 stabilized older Analyze, Database, Browser, and artifact surfaces, and it
confirmed v4.0 entry readiness. v4.0 should differ from v3.9 in the following
ways:

- v3.9 fixed practical gaps; v4.0 declares and verifies the stable public
  contract.
- v3.9 improved screen-level copy; v4.0 verifies cross-screen workflow
  continuity.
- v3.9 documented the localStorage inventory; v4.0 makes the local data contract
  an explicit release artifact.
- v3.9 allowed non-blocking public-quality notes; v4.0 should close or
  deliberately classify them before release.

## Scope Findings

### Already Complete

- Analyze preset and recent-analysis storage exist and have safe load fallback.
- Trainer recent, mistake, and filter storage use versioned keys and safe
  fallback.
- Database to Analyze handoff is explicit and does not auto-run analysis.
- Trainer reset flows preserve unrelated local records.
- Core/server/web/smoke verification is green at v3.9.
- Public sanitized release flow is proven through v3.9.

### Actual Gaps

- SECURITY.md contains a mojibake line in the reporting section and should be
  repaired with public-safe wording.
- `trainerOptions.ts` contains one mojibake fallback label for missing Trainer
  tree configuration.
- README and RELEASE_HISTORY in the original repo still describe the latest
  completed public tag as v3.8, while public GitHub is already at v3.9.
- GitHub Actions still reports a Node runtime deprecation annotation from
  official actions, even though project commands use Node.js 24 and pass.
- There is no single stable state model document tying Analyze, Database,
  Trainer, and verification states together.
- There is no single release acceptance checklist for v4.0.

### Out Of Scope

- New solver or data-generation engine.
- Server-side user history, accounts, login, or cloud sync.
- New DB/raw data bundle or generated artifact publication.
- Live/RTA/OCR/screen capture/overlay/hotkey/watcher/client integration.
- Large component rewrite or broad UI redesign.
- Unapproved dependency or package-lock changes.
