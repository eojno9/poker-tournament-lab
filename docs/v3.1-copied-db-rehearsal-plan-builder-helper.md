# v3.1 Copied DB Rehearsal Plan Builder Helper

## Purpose

The copied DB rehearsal plan builder helper creates a deterministic, in-memory plan shape for a future copied-DB-only import rehearsal. It combines the Step 2 approval contract evidence and the Step 3 rehearsal guard decision with validation, SHA, rollback, and report policy inputs.

This helper does not copy a DB, read a DB, write a DB, run an import, create report JSON, connect API/UI routes, or add package scripts.

## Non-goals

- No actual DB copy.
- No actual DB write.
- No production DB write.
- No DB schema migration.
- No product import route connection.
- No API/UI import flow.
- No `package.json` script.
- No raw HRC zip access, copy, extraction, or commit.
- No report JSON generation.
- No filesystem access.
- No DB access.
- No `process.env`, `Date.now`, or random value usage.

## Relationship to Approval and Guard Helpers

Step 2 decides whether a copied DB target approval contract is satisfied.

Step 3 decides whether target kind, target location, approval, DB SHA, validation, rollback, backup, privacy, and source archive DB injection checks are safe enough for a future rehearsal.

Step 4 turns those decisions into a plan:

- status
- dry-run-only flag
- future copied DB write rehearsal eligibility flag
- step list
- blocked reasons
- warnings
- required next checks

The plan builder does not execute any step.

## Plan Status

`READY_FOR_DRY_RUN` means the plan can remain in dry-run-only mode. No write-capable rehearsal is allowed.

`READY_FOR_COPIED_DB_WRITE_REHEARSAL` means the in-memory inputs show that a future copied DB write rehearsal could be considered. This still does not perform a write.

`BLOCKED` means one or more required checks failed or forbidden scope was requested.

## Dry-run-only Versus Copied DB Write Rehearsal

Dry-run-only requires:

- guard dry-run-only allowed
- approval dry-run allowed
- no write flags
- validation, SHA, rollback, and policy checks pass

Copied DB write rehearsal readiness requires:

- guard rehearsal allowed
- approval approved
- approval write allowed
- approval copied DB write allowed
- original DB SHA before and after present and unchanged
- copied DB SHA baseline present
- preview validation pass
- duplicate/canonical key validation pass
- rollback plan present and rollback verification required
- report file writing disabled

## Always-disabled Outputs

The plan output always keeps:

- `productionDbWriteAllowed: false`
- `reportFileWriteAllowed: false`

The helper reports readiness only. It does not enable actual file or DB writes.

## Plan Steps

Possible plan steps:

- `PRECHECK`
- `VALIDATE_PREVIEW`
- `VALIDATE_DUPLICATES`
- `VERIFY_ORIGINAL_DB_SHA`
- `VERIFY_COPIED_DB_SHA`
- `VERIFY_ROLLBACK_PLAN`
- `APPLY_TO_COPIED_DB_REHEARSAL`
- `POST_REHEARSAL_DIFF`
- `CLEANUP`

`APPLY_TO_COPIED_DB_REHEARSAL` appears only in a ready copied DB write rehearsal plan.

## Blocking Conditions

The plan is blocked by:

- failed guard decision
- production DB write request
- report file write request
- original DB SHA mismatch
- missing original DB SHA
- missing copied DB SHA baseline
- failed preview validation
- failed duplicate/canonical key validation
- validation blocked count above zero
- missing rollback plan
- missing rollback verification requirement
- neither dry-run-only nor copied DB rehearsal readiness
- approval not approved for copied DB rehearsal

## Warning Conditions

Warnings include:

- `importPreviewAllowed` is zero
- console summary is allowed while report file writing remains disabled
- propagated approval and guard warnings

## Privacy and Path Redaction

The plan output must not expose real local paths, private user tokens, emails, or raw HRC paths. Any private strings in IDs, target paths, warnings, blocked reasons, or next checks are redacted.
