# v4.0 plan

## Goal

v4.0 is the stable public milestone for the current off-table study product. It
should confirm that Analyze, Database, Trainer, local verification, public docs,
and sanitized release flow work together as a maintainable public repository.

## Non-Goals

v4.0 must not add:

- RTA or live assistance
- OCR, screen capture, overlays, hotkeys, watchers, or poker-client integration
- account, login, cloud sync, or server-side user history
- bundled DB, raw data, raw archive, or generated reports
- a new solver or data-generation engine
- large UI redesign or full app component split
- unapproved dependency or package-lock changes

## Must

1. Public status sync
   - Update original README and RELEASE_HISTORY from v3.8/v3.9-local wording to
     the completed v3.9 public release state.
   - Keep public-safe wording and avoid source/provider/permission details.

2. Public security copy repair
   - Replace the mojibake reporting line in SECURITY.md with readable
     public-safe Korean wording.
   - Keep the warning against public disclosure of secrets, paths, local data,
     and private contact details.

3. Korean-first copy repair
   - Fix the mojibake Trainer tree-config fallback label.
   - Scan touched UI for new English-only user-facing labels.

4. Unified workflow verification
   - Verify the flow: entry/navigation -> Database/Browser review -> Analyze
     handoff -> Analyze validation/result -> preset/recent reuse -> Trainer ->
     mistake review/local statistics.
   - Confirm each transition has clear context, no hidden auto-analysis, and
     safe reset behavior.

5. Stable state model
   - Document idle, ready, loading/processing, success, empty, invalid, not
     solved, unavailable, recoverable error, and completed states.
   - Tie each state to user copy, next action, data preservation, reset scope,
     and test expectation.

6. Local data contract
   - Freeze the v4.0 storage key inventory and compatibility expectations.
   - Confirm corrupt, partial, unknown, and write-failure fallback behavior.
   - Avoid unnecessary key/version bumps.

7. Release acceptance criteria
   - Define the v4.0 release checklist before implementation starts.
   - Include public-safe scans, CI, tests, artifacts, normal push, and tag
     conflict checks.
