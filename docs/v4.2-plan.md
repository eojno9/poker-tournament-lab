# v4.2 plan

## Goal

Close the highest-value maintenance gaps left after v4.1 while preserving the
stable Korean-first off-table study workflow and all public-safety boundaries.

## Non-goals

- no new solver or strategy generation engine
- no bundled DB, raw data, archives, or generated reports
- no server-side user history, account, login, or cloud synchronization
- no RTA, live assistance, OCR, screen capture, overlay, hotkey, watcher, or
  poker-client integration
- no broad App, Analyze, Browser, Import, or Database rewrite
- no new frontend, state, storage, logging, or observability framework
- no dependency or package-lock change
- no storage key rename or version bump
- no automatic commit, tag, remote mutation, or push

## Must

1. Extract the HRC artifact screen and its local rendering helpers from
   `App.tsx` without changing requests, DOM semantics, copy, accessible names,
   test IDs, or read-only behavior.
2. Replace raw global server exception messages with a generic public-safe
   response and an additive stable error code. Preserve existing successful
   responses and explicit validation behavior.
3. Add explicit `original` and `public` modes to the read-only release
   preflight. Original mode requires no remote; public mode validates the
   expected public remote/ref state without mutation.
4. Synchronize README and release-history current-status wording with the
   completed public v4.1 release before a future snapshot is prepared.
5. Add focused tests for each changed boundary.

## Should

- extract Playwright route setup and small navigation/storage/viewport helpers
  only where scenario intent becomes clearer
- retain all seven smoke scenarios and improve failure diagnostics
- add focused narrow-viewport checks for HRC artifact and one other dense view
- export the Database-to-Analyze handoff types and pure conversion helpers to
  a small module, without moving the Analyze screen
- document the approved public-safe docs subset in preflight output or review
  notes without copying internal audit documents

## Could

- provide both human-readable and JSON preflight summaries to stdout
- add a small shared key/value presentation primitive if HRC extraction would
  otherwise create circular or reverse imports
- add a stable server error-code type shared only where it removes duplication

## Implementation result

The maintenance bundle implements every Must boundary without broadening the
product scope:

- `HrcArtifactView.tsx` now owns HRC list/detail state, GET-only requests,
  filters, loading, empty, error, detail, and sanitized JSON presentation.
- `App.tsx` retains only the tab switch and component mount for HRC artifacts.
- unhandled server exceptions return a fixed Korean message and
  `INTERNAL_SERVER_ERROR`; explicit invalid and missing responses retain their
  existing message/status and add a stable code.
- browser API failures prefer an allowlisted server code and otherwise use the
  existing status-based Korean fallback. Raw response text is never rendered.
- release preflight now requires `--mode original` or `--mode public`, supports
  optional JSON stdout, and keeps all checks read-only.
- README status names v4.1 as the completed public release and v4.2 as local
  maintenance verification only.
- the existing HRC smoke scenario now checks a 390 px viewport, document-level
  overflow, visible controls, and table-local horizontal scrolling.
