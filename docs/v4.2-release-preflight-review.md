# v4.2 release preflight review

## Implemented behavior

The v4.2 script validates these common gates in both explicit modes:

- clean Git status
- clean package and lock files
- no tracked forbidden artifacts
- privacy/path/email, secret, and mojibake scans
- README and RELEASE_HISTORY current-tag wording
- SECURITY reporting guidance
- no conflicting local candidate tag

It is read-only and has `28` checks covering pure positive/negative states,
argument parsing, JSON output, clean local Git fixtures, and before/after
non-destructive snapshots.

## Closed gap

The state model has a single `remoteAbsent` boolean and labels that check
`original remote absent`. A fresh public upload clone correctly has an origin,
so the complete verdict cannot run there. Release operators must call lower
level scan functions or repeat manual checks.

The explicit mode model closes this gap without weakening the original-repo
remote policy.

## v4.2 mode design

### Original mode

- remote must be absent
- candidate local tag must be absent before tag creation
- working tree and package files must be clean
- current public status still names the last completed public tag
- no public ref mutation or network write

Usage:

```powershell
node scripts/public-release-preflight.mjs --mode original --candidate-tag v4.2 --expected-current-public-tag v4.1
```

### Public mode

- origin must match an explicit expected URL
- branch must match the expected branch
- remote main and candidate tag are read-only checks
- candidate remote tag must be absent before release
- current public status wording is checked for the intended snapshot phase
- package, artifact, privacy, secret, encoding, and SECURITY gates remain

Public mode may use `git ls-remote` because it is read-only. It must not call
fetch, clone, commit, tag, remote mutation, or push.

Usage:

```powershell
node scripts/public-release-preflight.mjs --mode public --candidate-tag v4.2 --expected-current-public-tag v4.1
```

Public mode additionally requires branch `main`, the expected origin, a
readable remote main, a local fast-forward relationship, no candidate remote
tag, and no tracked internal upload/audit policy document from the approved
exclusion set. `--expected-origin` is available for isolated local fixtures;
the default is the public project URL.

## Output

Human-readable output remains the default. Optional JSON may be emitted to
stdout for machine inspection. The script must not create a JSON report file.

Use `--json` to select the machine-readable stdout summary. Missing or unknown
mode values and malformed tags exit non-zero.

Each result includes:

- mode
- branch and HEAD
- candidate/current tag inputs
- named checks
- aggregate PASS/FAIL

Sensitive matches are reported as counts and categories, not literal values.

## False-positive controls

- public-safe placeholders remain allowed
- internal audit documents remain outside the public text roots unless
  explicitly approved
- examples use `example.test`, `<local-user>`, and similar placeholders
- a candidate tag string is evaluated only in the intended current-status
  section, not rejected everywhere in historical docs

## Read-only proof

Positive and negative mode tests compare before/after:

- HEAD
- porcelain status
- refs
- remotes

The test command also scans the script for prohibited mutation commands. A
failed gate exits non-zero without changing repository state.

The integration fixtures confirm clean original and public PASS paths using
temporary local Git repositories. They also confirm missing/unknown mode
failures and compare HEAD, status, refs, and remotes before and after each
positive execution.

## Closeout verification result

- actual clean original repository, human output: PASS
- actual clean original repository, JSON output: PASS and parseable
- isolated public clone with local bare remote: PASS
- wrong/missing public origin and non-fast-forward states: FAIL as intended
- dirty/package/artifact/status/tag and public-doc gates: FAIL as intended
- missing and unknown mode: non-zero as intended
- fixture total: `28` checks, PASS
- HEAD, status, refs, and remotes before/after: unchanged
- public remote `v4.2` at review time: absent

No GitHub write, fetch, tag creation, report file, or automatic document edit
is performed by either mode.

## Actions and status hotfixes

Actions remains a post-push verification step outside the script. Durable
snapshot text must not claim a future Actions result. A main-only status hotfix
is reserved for an actual stale or incorrectly rendered status; it never moves
the release tag.
