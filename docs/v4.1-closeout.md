## TrainerView Boundary

`TrainerView.tsx` owns Trainer-specific loading, session state, filters,
grading, duplicate-submit protection, mistake review, retry flow, local
statistics, storage notices, and fallback rendering. `App.tsx` imports and
renders that component only for the Trainer tab; it does not retain duplicate
Trainer markup or import Trainer internals back from the extracted view.

The extraction reduced `App.tsx` from approximately 5,507 lines to 4,624
lines. The extracted view is 919 lines and forms one coherent maintenance
boundary rather than a broad application rewrite. No reverse import to
`App.tsx` or circular dependency was found.

Regression verification covers:

- session start, progress, completion, and reset
- current/total question progress
- filter save, restore, and reset
- answer grading and duplicate-submit prevention
- unresolved, resolved, and dismissed mistake filters
- retry flow, session statistics, and all-local statistics
- `제공되지 않음` fallback copy
- Korean-first labels, accessible names, responsive baseline, and safety copy

The established seven-scenario smoke suite passed without reducing coverage.

## Safe Storage Contract

`safeStorage.ts` provides one exception boundary for storage availability and
`getItem`, `setItem`, and optional `removeItem`. Reads return an explicit
success/failure union; writes and removals return booleans. Raw storage
exceptions do not propagate through these helpers.

Domain parsing, shape validation, normalization, migration, record limits,
and reset meaning remain in their owning modules. Existing keys are unchanged:

| Domain | Key/version |
| --- | --- |
| Analyze presets | `poker-tournament-lab:analyze-presets:v1` |
| Recent analyses | `poker-tournament-lab:recent-analyses:v1` |
| Trainer recent | `ptl.trainer.recent.v1` |
| Trainer legacy recent | `ptl.trainer.recentAttempts.v1` |
| Trainer mistakes | `ptl.trainer.mistakes.v1` |
| Trainer filters | `ptl.trainer.filters.v1`, payload version `1` |

Tests confirm successful/null/throwing reads, successful/throwing writes and
removals, unavailable storage, corrupt and unknown payload fallback, and
legacy recent-history migration. Write/reset failure does not stop the current
Trainer or Analyze flow and does not delete unrelated local records.

Storage verdict: `SAFE_STORAGE_CONTRACT_PASS`

## API Error Mapping

The API request layer no longer reads a failed response body into an exception
message. Statuses map to Korean-first public messages for invalid requests,
authorization, not found, conflict, delay, unavailable service, and server
failure. Network and invalid-response failures have separate recovery copy,
and the explicit not-solved message remains distinct from a server failure.

UI call sites render mapped messages or caller-owned safe fallbacks. Tests
confirm that HTML, raw transport detail, stack-like detail, and path-like text
are not included in the rendered mapping. Successful response parsing and the
existing API contracts remain unchanged.

API verdict: `API_ERROR_MAPPING_PASS`
