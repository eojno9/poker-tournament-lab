## Opt-In Local Artifact Export Concept

The v2.7 export flow should be explicit and local.

Planned behavior:

- The user provides a raw HRC zip path.
- The raw HRC zip remains outside the repo.
- The command/script reads only the required zip entries in memory.
- The command/script builds the normal dry-run report.
- The command/script converts the dry-run report into the v2.6 artifact-safe schema.
- The command/script writes sanitized JSON under `artifacts/hrc-dry-run-reports/`.
- The command/script may also write comparison and index JSON files.
- The command/script does not commit generated artifacts.
- The command/script does not call product import routes.
- The command/script does not write to the database.
- The command/script does not call APIs.
- The command/script does not invoke UI code.

Default behavior should remain dry-run oriented. Any disk write to repo artifacts must be an explicit local command/script action, not an automatic side effect of tests, build, smoke, app runtime, server startup, or UI runtime.

## Export Command/Script Design Direction

The first implementation should prefer a local script/helper boundary rather than product runtime integration.

Candidate direction:

- A local script or test-scoped command accepts:
  - raw HRC zip path
  - optional output directory
  - optional comparison/index mode
- The default output directory is `artifacts/hrc-dry-run-reports/`.
- The script validates that the raw input path is outside generated artifacts and is not copied into the repo.
- The script runs the existing dry-run reader.
- The script builds an artifact-safe report.
- The script writes only sanitized JSON outputs.
- The script prints a concise summary with output paths and safety flags.

The script must not be imported by server routes, web runtime code, database writers, or product import logic.

## Output Path And File Naming Policy

Planned output directory:

- `artifacts/hrc-dry-run-reports/`

Planned file names:

- report: `hrc-dry-run-YYYYMMDD-HHMMSS-<sanitized-name>.json`
- comparison summary: `hrc-dry-run-comparison-YYYYMMDD-HHMMSS.json`
- index: `hrc-dry-run-index-YYYYMMDD-HHMMSS.json`

Rules:

- Use timestamp-based names.
- Include a sanitized raw zip filename hint only when safe.
- Never include a full local path.
- Never include drive letters, user directories, account tokens, or email-like text.
- Strip path separators.
- Normalize spaces, non-ASCII, and special characters into safe filename tokens.
- Use `.json` only.
- Defend against path traversal such as `../` and absolute output file names.
- Ensure writer output stays inside the allowed artifact output directory.

## Artifact Commit Policy

Generated artifact JSON files are diagnostic outputs and must not be automatically committed.

Policy:

- Generated sanitized reports may be reviewed by the user.
- The decision to commit any generated artifact remains manual and separate.
- Raw HRC zip originals must never be committed.
- Full raw zip extraction output must never be committed.
- Export scripts must not run `git add` or `git commit`.
- Release or backup steps must explicitly check that raw zip originals are not included.

## Safety And Privacy Policy

v2.7 must preserve the safety guarantees from v2.5 and v2.6.

Rules:

- Raw HRC zip files stay outside the repo.
- Raw HRC zip originals are never copied into artifacts.
- Full raw zip extraction output is never stored.
- Report JSON must not include raw path clear text.
- Report JSON must not include `C:\Users\` clear text.
- Report JSON must not include user tokens such as local account names.
- Report JSON must not include email addresses.
- Privacy warning entries store category names only, not raw values.
- Long hand strategy payloads are not stored.
- Mismatch samples remain capped at 3 entries.
- Report JSON must not be treated as a product import payload.
- Report JSON must not be automatically promoted to production DB.

## Amount Semantics Policy

HRC raw amount values remain uninterpreted in v2.7.

Policy:

- `amountUnit: UNKNOWN`
- `amountInterpretation: RAW_HRC_AMOUNT_UNINTERPRETED`
- `bbConversionApplied: false`
- `chipConversionApplied: false`
- Raw amount values may be preserved as raw labels or source metadata summaries.
- Do not infer bb/chip units until HRC format documentation or multiple verified samples support it.

## Multiple Node Policy

v2.7 keeps the v2.5/v2.6 multiple node policy.

Rules:

- Prefer `nodes/0.json` when present.
- If `nodes/0.json` is absent, select the first `nodes/*.json` by deterministic string sort.
- Record `selectedNodeEntry`.
- Record `selectedNodeReason`.
- Record `multipleNodeEntriesDetected`.
- Keep `multiNodeAggregationApplied: false`.
- Do not combine multiple nodes into one product candidate.
- Multi-node aggregation remains future work.
