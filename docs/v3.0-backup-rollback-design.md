# v3.0 Backup / Rollback Design

## 1. Purpose

Before any future product DB import becomes DB write-capable, v3.0 defines the policy for pre-import backup, post-import verification, and rollback on failure.

This step is docs-only. It does not create a backup, copy the DB, read or write the DB, run restore rehearsal, implement an import command, connect product import routes, or add API/UI/runtime behavior.

## 3. Required Gates Before Any Future DB Write

Future DB write work must not proceed unless all gates pass:

- Explicit user approval.
- `git status` clean.
- `npm.cmd run test` PASS.
- `npm.cmd run build` PASS.
- `npm.cmd run test:smoke` PASS.
- Current DB SHA256 recorded.
- Current verification report recorded.
- Privacy/path scan PASS.
- Raw zip absence confirmed.
- Generated artifact JSON absence confirmed.
- `artifacts/hrc-dry-run-reports` absence confirmed.
- Product import preview validation PASS.
- Duplicate/canonical key preview PASS.
- Backup folder plan confirmed.
- DB backup manifest plan confirmed.
- Rollback instructions written.
- Restore rehearsal plan written.

## 4. Backup Folder Policy

Future backup folders must live outside the repo.

Expected local format:

```text
<local-backup-root>/v3.0-import-backup-YYYYMMDD-HHMMSS
```

Policy:

- Do not put actual user-local paths in public-facing docs.
- Use `<local-backup-root>` in GitHub/public-safe documents.
- Do not include raw HRC zip files in product import backups.
- Do not include generated dry-run artifacts in product import backups unless a later step explicitly approves sanitized artifact handling.
- Do not create backup folders in this Step 5 docs-only work.

## 5. Backup Manifest Policy

Future backup manifests should record:

- Timestamp.
- Branch.
- Commit hash.
- DB SHA256 before import.
- DB file name.
- Import preview summary.
- Duplicate/canonical key preview summary.
- Source classification summary.
- Raw zip absence confirmation.
- Generated artifact absence confirmation.
- Privacy/path scan result.
- Restore rehearsal plan.
- Rollback instructions.

## 6. Rollback Policy

If a future DB write fails or post-import verification fails:

- Stop the import process.
- Determine whether app/server processes need to stop before rollback.
- Check DB sidecar files: `*.db-shm`, `*.db-wal`.
- Decide whether to preserve the failed DB for audit.
- Restore the backup DB to the expected DB location.
- Confirm post-rollback DB SHA256.
- Re-run `npm.cmd run test`.
- Re-run `npm.cmd run build`.
- Re-run `npm.cmd run test:smoke`.
- Recheck existing verification metrics.
- Write a rollback report.

This policy is descriptive only in Step 5. No restore is executed here.

## 7. Post-import Verification Policy

Future imports may increase total solution count, but existing stability metrics must not regress.

Baseline metrics:

- exact lookup `262/262`
- random lookup `20/20`
- duplicate canonical key `0`
- near-match HRC false positives `0`
- forbiddenHrcHit true count `0`

Policy:

- Existing lookup behavior must not regress.
- Duplicate canonical key count must remain `0`.
- Near-match false positives must remain `0`.
- Any metric regression blocks promotion until investigated.

## 8. Import Approval Gate

Future DB write is not allowed without an explicit approval gate.

Requirements:

- Default mode must be dry-run.
- Write mode requires explicit user approval.
- Write mode cannot be enabled accidentally by UI.
- Write mode cannot be enabled by a GET request.
- Write mode must record backup id.
- Write mode must record import batch id.
- Write mode must record source provenance.
- Write mode must be reversible by documented rollback process.
