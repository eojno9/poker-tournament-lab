# v3.3 no-write safety gate plan

## Scope

This document defines the no-write safety gate requirements for the future copied DB rehearsal command.

This is a planning-only, docs-only step.

No guard code, CLI code, runtime code, API code, UI code, dependency change, DB access, raw data access, or generated artifact creation is performed in this step.

## Safety principle

The future command must fail closed.

A dry-run command is safe only when it refuses all write, production, raw data, generated artifact, and live-assistance behaviors by default.

## Required invariants

| Invariant | Required behavior |
| --- | --- |
| Explicit dry-run | Command must fail if `--dry-run` is missing |
| No write mode | Command must reject write/import/apply/migrate flags |
| No production target | Command must reject production-like target names or paths |
| No raw DB | Command must reject raw DB file paths |
| No raw zip | Command must reject archive inputs |
| No generated artifact by default | Command must print console-only report |
| No API/UI connection | Command must not connect to API routes or Web UI |
| No live assistance | Command must not support live play or poker client integration |
| No sensitive wording | Output must not reveal source/provider/permission/private paths |

## Forbidden input patterns

Document categories, not private concrete paths.

Examples:

- production-like target labels
- raw database file extensions
- archive file extensions
- import/apply/write/migrate flags
- generated artifact output paths
- API/UI mode flags
- live watcher / OCR / screen capture / overlay / hotkey wording

Do not include private paths or specific dataset/source/provider names.

## Fail-closed behavior

For any unsafe input, the future command should:

1. refuse to continue
2. avoid opening files
3. avoid writing files
4. avoid creating reports
5. print a public-safe explanation
6. exit with non-zero status

The command should evaluate safety before any future file read is considered. If target classification cannot be completed safely from provided metadata, the result should be blocked.

## Public-safe refusal wording

Use neutral language only.

Allowed examples:

- `Refused: dry-run mode is required.`
- `Refused: production targets are not supported.`
- `Refused: raw data inputs are not supported.`
- `Refused: write operations are not available for this command.`
- `Refused: this tool is for off-table study only.`

Do not mention private source/provider/dataset names.

## Verification requirements for future implementation

Future implementation must verify:

- package and lockfile changes are separately approved
- dry-run tests pass
- fail-closed tests pass
- no DB/raw fixtures are added
- no generated artifacts are committed
- no API/UI integration is added
- no RTA/live scope is introduced

Additional future checks should include:

- output redaction checks
- no report file creation by default
- no sidecar file creation
- no package script unless separately approved
- no public wording that reveals private data origin details

`NO_WRITE_GATE_PLAN_READY`
