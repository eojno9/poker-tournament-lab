# v4.0 local data contract

## Purpose

v4.0 keeps browser-local data as an explicit public contract. No server-side user
history, account sync, cloud storage, or bundled data artifact is introduced.

## Contract Rules

- Data remains in the user's browser.
- Values are derived from user-entered form state, local study workflow state,
  or locally configured study rows.
- Storage must not contain raw local paths, private credentials, source/provider
  details, or permission history.
- Corrupt, partial, unknown, or incompatible payloads must fall back safely.
- Write failures should keep the current session usable.
- Clear/reset actions must be scoped:
  - recent analysis clear does not clear Analyze presets
  - Trainer recent clear does not clear Trainer mistakes unless separately
    requested
  - Trainer filter clear does not clear Trainer history
  - Database to Analyze handoff reset does not clear presets or recent analyses

## Payload Expectations

### Analyze Presets

- Envelope shape: `{ version, presets }`
- Entries contain id, name, timestamps, version, and cloned Analyze form state.
- Broken JSON or invalid candidate shapes load as an empty list.
- Save requires a non-empty preset name.

### Recent Analyses

- Array shape with a maximum of 20 entries.
- Entries contain form state, result source, source label, summary, and sanitized
  metadata.
- Broken JSON or non-array payloads load as an empty list.

### Trainer Recent

- Array shape with a maximum of 30 entries.
- Current key is `ptl.trainer.recent.v1`.
- Legacy key `ptl.trainer.recentAttempts.v1` may migrate into the current key
  when the current key is absent.
- Broken or non-array payloads reset to an empty safe fallback.

### Trainer Mistakes

- Array shape with a maximum of 50 entries.
- Status values: `unresolved`, `resolved`, `dismissed`.
- Repeated incorrect attempts update unresolved mistake records.
- Correct retry can mark a matching mistake as resolved.
- Dismiss action hides the item without clearing the queue.

### Trainer Filters

- Envelope includes `version: 1`, filters, hand input, and seed input.
- Unknown version loads defaults without breaking Trainer initialization.
- Corrupt or non-object payload resets to default filters.
- Write/clear failures return a safe result and keep the app usable.

## Migration Policy

v4.0 should not introduce new storage keys or version bumps unless implementation
finds a real compatibility issue. If a migration is needed, it must:

- preserve existing valid v1 data
- reject unknown shapes safely
- avoid broad rewrites
- include focused tests
- document reset behavior

## Release Checks

- Storage tests pass.
- Smoke verifies existing Analyze/Trainer workflows.
- No server persistence is added.
- No package or dependency change is needed for storage handling.

## Implementation Alignment Result

The v4.0 implementation bundle did not change storage keys, payload versions,
or persistence scope.

Confirmed code alignment:

- Analyze presets use `poker-tournament-lab:analyze-presets:v1` with an
  envelope `{ version, presets }` and safe load fallback.
- Recent analyses use `poker-tournament-lab:recent-analyses:v1` with a capped
  array and safe load fallback.
- Trainer recent history uses `ptl.trainer.recent.v1` and keeps legacy migration
  from `ptl.trainer.recentAttempts.v1`.
- Trainer mistakes use `ptl.trainer.mistakes.v1` with unresolved, resolved, and
  dismissed status handling.
- Trainer filters use `ptl.trainer.filters.v1` with `version: 1`, safe default
  fallback, and write-failure tolerance.

No server persistence, account sync, cloud storage, key rename, version bump, or
large storage rewrite was introduced.
