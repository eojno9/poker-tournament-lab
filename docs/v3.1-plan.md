# v3.1 Import Rehearsal Planning

## Summary

v3.1 starts from the completed v3.0 stable baseline and plans the next stage: copied-DB-only import rehearsal design. This branch does not implement product import, does not connect API or UI import flows, and does not write to the production database.

## v3.0 Stable Baseline

- Stable tag: `v3.0`
- v3.0 tag target: `4b6848351ba4dffb4301d302dfd076447267875f`
- v3.0 stable restore rehearsal: `STABLE_RESTORE_REHEARSAL_PASS`
- DB SHA256 baseline: `92BBF8DA75EA34ABF4CBCF8111ADBCD729DA93D7DDE690E14EB52482A8E16461`
- v3.0 scope: no-write product import foundation
- Known issue at v3.0 stable: none

## Goals

- Design a copied DB based real import rehearsal flow.
- Evaluate write behavior only against an explicit copied DB, never the original production DB.
- Preserve the explicit approval gate before any write-capable rehearsal can run.
- Define rollback verification requirements before and after a copied DB rehearsal.
- Preserve the source archive DB injection policy used during restore rehearsals.
- Keep privacy/path scan and raw artifact absence checks as release gates.
- Define evidence required before any later product import route, API, UI, or package script exposure.

## Non-goals

- No original production DB write.
- No production DB schema migration.
- No product import route connection.
- No API import flow.
- No UI import flow.
- No immediate `package.json` import or preview script.
- No raw HRC zip direct processing.
- No raw HRC zip copy, extraction, or commit.
- No generated artifact JSON commit.
- No GitHub push.
- No change to v3.0, v3.0-rc2, v3.0-rc1, v2.9, v2.9-rc2, or v2.9-rc1 tags.

## Safety Invariants

- Production DB write remains forbidden.
- Production DB SHA256 baseline must remain unchanged unless a later step explicitly approves a copied DB rehearsal that does not touch the production DB.
- Copied DB target must pass path guard checks.
- Any write-capable rehearsal must require explicit approval and must target only a copied DB under an approved restore/rehearsal area.
- Product import route remains disconnected.
- API/UI import surfaces remain absent unless separately planned and approved.
- `package.json` import/preview scripts remain absent unless separately planned and approved.
- Raw HRC zip files remain repo-external and are not copied or extracted.
- `artifacts/hrc-dry-run-reports` remains absent unless a later step explicitly approves sanitized artifact use.
- Privacy/path scan must report zero forbidden real local path, user token, or email hits.
- GitHub push remains out of scope for local planning work.

## Source Archive DB Injection Policy

The v3.0 restore policy remains active:

- Source archive restore verifies file layout, privacy scan, DB presence, and DB SHA.
- Git bundle clone verifies Git-tracked source, tags, checkout, and source behavior.
- Because the SQLite DB may be untracked and absent from a bundle clone, DB-dependent verification may inject the verified source archive DB into a restore-test bundle clone.
- Any DB injection must happen only inside a restore-test or copied-DB rehearsal folder.
- DB sidecar files must not be copied as rehearsal inputs.

### Copied DB Rehearsal Guard Helper

Status: completed in `docs/v3.1-copied-db-rehearsal-guard-helper.md`.

The Step 3 helper combines target kind, target location, Step 2 approval contract evidence, DB SHA evidence, preview validation, duplicate/canonical key validation, rollback evidence, backup evidence, privacy scan, and source archive DB injection policy acknowledgment. The helper remains pure and does not create a copied DB, copy files, read a DB, write a DB, run an import, connect a route, add API/UI flow, add a package script, or generate report JSON.

Guard pass requires:

- `targetKind: COPIED_DB`
- `targetLocationKind: RESTORE_TEST` or `BACKUP_ROOT_REHEARSAL`
- explicit approval evidence for future copied DB rehearsal
- original DB SHA before/after present and unchanged
- copied DB SHA baseline present
- preview validation pass
- duplicate/canonical key validation pass
- rollback plan provided
- privacy/path scan pass
- backup manifest available
- source archive DB injection policy acknowledged

Guard reject conditions include production DB target, repo-local target, unknown target, unsafe copied DB location, production DB write approval, missing approval, invalid approval, DB SHA mismatch, missing rollback plan, failed preview/duplicate/privacy checks, missing backup manifest, and missing source archive DB injection policy acknowledgment.

The helper can also mark a safe dry-run-only path with `dryRunOnlyAllowed: true` when write flags are false. Even when guard conditions pass for a future rehearsal, output write flags remain false in this step.
