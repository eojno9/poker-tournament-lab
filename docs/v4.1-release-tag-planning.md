# v4.1 release and tag planning

## Candidate

| Item | Decision |
| --- | --- |
| Candidate tag | `v4.1` |
| Candidate title | `v4.1 - Maintenance safety and structure` |
| Original target | The docs-only closeout commit produced by this bundle |
| Tag type | Annotated |
| Public release method | Fresh sanitized source snapshot |

A read-only remote check during closeout confirmed that public `main` is
accessible and remote `v4.1` is absent. The release bundle must repeat this
check immediately before creating and pushing the public tag.

The exact original target SHA is recorded in the post-commit completion
report. The implementation commit alone is not the target because the
candidate includes this verified closeout and release plan.

## Original Repository Step

After explicit release execution begins:

1. confirm the original working tree is clean
2. confirm the original repository still has no remote
3. confirm local `v4.1` does not exist
4. create one annotated `v4.1` tag at the closeout commit
5. do not push from the original repository

Suggested message:

```text
v4.1 - Maintenance safety and structure
```

## Fresh Sanitized Snapshot

Create a fresh folder outside the original repository from the current public
GitHub `main`, then apply the reviewed `v4.1` tree snapshot without copying the
original `.git` history.

The public copy must exclude dependencies, build/test outputs, local data,
archives, generated reports, environment files, and internal audit documents
that contain scan-target literals. Include only the reviewed public-safe docs
subset.

Before its snapshot commit, run:

- typecheck, tests, build, and seven-scenario smoke
- the read-only public release preflight with expected current public tag
  `v4.0`
- tracked-artifact and public-safe scans
- README and RELEASE_HISTORY status review

Commit the sanitized snapshot with a public-safe maintenance message.

## Public Tag And Push

Create an annotated public `v4.1` tag at the sanitized public snapshot commit.
The public tag target will differ from the original target by design.

Proceed only when:

- the public folder is clean on `main`
- the remote main is the expected fast-forward parent
- remote `v4.1` does not exist
- all local verification and scans pass
- no force push or tag overwrite is required

Push only:

```powershell
git push origin main
git push origin v4.1
```

Do not use `git push --tags`.

## Post-Push Verification

Confirm:

- remote `main` equals the public snapshot commit
- remote annotated `v4.1` peels to that snapshot commit
- GitHub Actions typecheck/test/build and Playwright smoke succeed
- rendered/raw README identifies `v4.1` as the completed public tag
- RELEASE_HISTORY records the completed public main/tag state
- public-safe and artifact blockers remain zero

If status text needs a post-push correction, create one docs-only main commit
and normal-push `main` only. Do not move, delete, recreate, overwrite, or
re-push the `v4.1` tag.

## Hard Stops

Stop when any of the following occurs:

- original repository remote exists
- remote `v4.1` already exists
- fast-forward is not possible
- force push or tag overwrite would be required
- package or lockfile changes unexpectedly
- a dependency, forbidden artifact, or public-safe blocker appears
- typecheck, tests, build, smoke, or preflight fails
- README and RELEASE_HISTORY disagree
