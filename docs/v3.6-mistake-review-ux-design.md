# v3.6 mistake review UX design

## Scope

This document defines the v3.6 mistake review UX refinement plan for the
Korean-first, off-table Trainer.

This is a docs-only design step. It does not implement code, change
dependencies, access DB/raw data, create generated artifacts, push to GitHub,
or create/change tags.

## v3.5 Mistake Review Baseline

v3.5 introduced local-only mistake review as part of the Trainer UX:

- incorrect answers can create local mistake records
- mistakes can be retried off-table
- correct retry can mark a mistake as resolved
- items can be dismissed
- recent attempts and mistake state stay in browser localStorage
- local statistics can summarize mistake counts and accuracy
- safety copy keeps the Trainer framed as off-table study only

The v3.5 design defined the core model and statistics. v3.6 should refine the
visible review experience without changing the persistence boundary.

## v3.6 Improvement Goals

Primary goals:

- make the mistake review section easier to understand
- make unresolved, resolved, and hidden/dismissed states clearer
- make the retry flow more obvious
- improve empty states for no mistakes and all-resolved queues
- make clear/reset copy safer and less alarming
- decide whether retry count and recent review time should be visible
- clarify how local statistics should reflect the mistake queue
- preserve Korean-first, off-table-only copy

## Korean-First Mistake Review Copy Criteria

Future implementation should follow these copy rules:

| Area | Criteria |
| --- | --- |
| Section title | Korean-first, clear that this is review, not live advice |
| Queue labels | Korean-first labels for unresolved, resolved, and hidden/dismissed states |
| Retry action | Korean-first button text that implies off-table retry only |
| Empty states | Korean-first, calm, and actionable |
| Clear/reset copy | Explicitly says local records only |
| Statistics labels | Korean-first and scoped to local Trainer attempts |
| Safety note | Concise off-table/no-live-assistance wording |

Candidate label intent:

- mistake review section title
- unresolved mistakes
- resolved mistakes
- hidden or dismissed mistakes
- retry the spot
- clear local review records
- no mistakes yet
- all mistakes resolved

The exact Korean labels should be selected during implementation and verified
with the Korean-first copy audit.

## UX Improvement Candidates

### Mistake Review Section Structure

Candidate structure:

- Korean-first mistake review header
- compact count summary
- state filter or segmented control for unresolved/resolved/hidden
- selected item preview
- retry action
- local-only note

Expected effect:

- users can see why the section exists
- unresolved work is visible first
- resolved/hidden items do not dominate the screen

### Unresolved, Resolved, And Hidden State Display

Candidate behavior:

| State | Display intent |
| --- | --- |
| Unresolved | primary queue, retry available |
| Resolved | secondary, collapsed or filtered by default |
| Hidden/dismissed | not shown by default; available only if a simple filter exists |
| Unsupported retry | visible but disabled with clear Korean copy |

Rules:

- unresolved items should be the default view
- resolved items should not feel like new tasks
- hidden/dismissed items should not silently disappear without clear copy
- state labels should not imply server sync or account storage

### Retry Button And Flow

Candidate retry flow:

1. User opens mistake review
2. User selects an unresolved local mistake
3. User clicks the Korean-first retry action
4. Trainer shows the same off-table study spot
5. Feedback is shown after answer
6. Correct answer marks the item resolved
7. Incorrect answer keeps it unresolved and increments retry count

Copy guidance:

- retry wording should imply off-table retry only
- after retry, feedback should explain the local state change
- if retry is unavailable, explain that the study spot cannot be restored locally

### Empty States

Required empty states:

| State | Candidate copy intent |
| --- | --- |
| No mistakes yet | Mistakes appear here after incorrect Trainer answers |
| All resolved | All local mistakes are currently resolved |
| Hidden only | Hidden items are excluded from the main review queue |
| Corrupt local data fallback | Local review records were reset safely |
| Unsupported item | This saved item cannot be retried in the current local dataset |

The copy should be calm and practical, not celebratory in a way that obscures
what the user can do next.

### Clear Mistake Records Copy

Candidate copy rules:

- say this clears local review records on this device
- do not imply server deletion
- do not imply data source deletion
- require a clear action label
- avoid destructive-looking UI unless confirmation is added

Preferred wording direction:

- clear only local mistake records on this device
- clear local review records
- do not change local study data or configured source data

### Retry Count And Recent Review Time

Decision:

- retry count should be visible when it helps prioritize repeated mistakes
- recent review time may be useful but should not clutter the main queue
- exact timestamps are optional; relative or compact Korean copy is preferred
- no server time or account time is introduced

Candidate display:

| Field | Default visibility | Reason |
| --- | --- | --- |
| `retryCount` | Should show in item detail or compact chip | Helps identify repeated weak spots |
| `lastReviewedAt` | Could show in item detail | Useful, but lower priority |
| `firstWrongAt` | Hidden by default | Too much detail for the queue |

### Local Statistics Connection

The mistake queue should connect to local statistics without becoming a separate
analytics system.

Candidate statistics:

- unresolved mistake count
- resolved mistake count
- retry count total or average
- recent mistake trend from local attempts
- optional by-position mistake count in the later statistics step

Rules:

- derive from local attempts and local mistake records
- denominator-0 states must be safe
- no server persistence
- no account/cloud sync
- no data source disclosure

## Implementation Priority

### Must

- make unresolved mistake queue the default focus
- improve no-mistakes empty state
- improve all-resolved empty state
- clarify retry action and retry outcome copy
- clarify local-only clear/reset copy
- preserve unresolved/resolved/dismissed states
- keep localStorage-only persistence
- avoid package/dependency changes
- avoid API/server persistence

### Should

- add clearer unresolved/resolved/hidden counts
- show retry count in item detail or compact metadata
- make hidden/dismissed state discoverable without clutter
- keep resolved items secondary or collapsible
- update smoke tests if visible labels change
- verify corrupt localStorage fallback copy

### Could

- show recent review time in item detail
- add compact queue filters
- add microcopy for unsupported retry
- add a local-only clear confirmation
- add screenshot/manual review notes

## Non-Goals

This design does not include:

- server persistence
- account or cloud storage
- DB/raw data access
- generated report creation
- live/RTA behavior
- OCR, screen capture, overlay, hotkey, watcher, or poker client integration
- new dependencies
- package-lock changes
- public upload or tag operations
- detailed position/action statistics implementation
- localStorage key migration implementation

## localStorage And Data Boundary

The mistake review UX must continue to use local-only browser storage.

Expected keys remain versioned, such as:

- `ptl.trainer.mistakes.v1`
- `ptl.trainer.recentAttempts.v1`

Rules:

- corrupt JSON must fall back safely
- clear/reset affects local review records only
- no network write is introduced
- no server-side Trainer history is introduced
- no raw path/source/provider/permission metadata is stored or shown

## Test Plan

Future implementation should verify:

| Check | Expected result |
| --- | --- |
| No mistakes state | Korean-first empty state, no confusing action |
| Unresolved display | unresolved mistakes are visible by default |
| Resolved display | resolved items are clearly secondary |
| Hidden/dismissed handling | dismissed items do not clutter the main queue |
| Retry flow | retry restores the local study spot and updates state |
| Correct retry | item becomes resolved |
| Incorrect retry | item remains unresolved and retry count increments |
| Clear local records | only local mistake records are cleared |
| localStorage fallback | corrupt local data resets safely |
| Korean copy | touched labels remain Korean-first |
| Safety copy | off-table/no-live-assistance boundary remains visible |

Recommended verification command set for implementation:

```powershell
npm.cmd run typecheck
npm.cmd run test
npm.cmd run build
npm.cmd run test:smoke
```

## Acceptance Criteria For Implementation

An implementation step can be considered ready when:

- unresolved/resolved/dismissed states are understandable
- retry flow is clear and off-table only
- empty states are Korean-first and useful
- clear/reset copy says local-only
- local statistics remain derived from local data
- no server/API persistence is introduced
- no DB/raw/generated artifact access exists
- no live-use, RTA, OCR, screen capture, overlay, hotkey, watcher, or poker client scope exists
- tests/build/smoke pass
