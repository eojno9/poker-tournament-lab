# v3.1 Copied DB Rehearsal Guard Helper

## Purpose

The copied DB rehearsal guard helper decides whether a future copied-DB-only import rehearsal target is safe enough to proceed as a rehearsal candidate. It is a pure helper and does not create a copied DB, copy files, read a DB, write a DB, run an import, connect routes, or add package scripts.

The helper is built on the Step 2 copied DB target approval contract. It can consume an approval contract result through an adapter and combine that approval evidence with target kind, target location, DB SHA evidence, preview validation, duplicate validation, rollback evidence, backup evidence, privacy scan, and source archive DB injection policy acknowledgment.

## Non-goals

- No actual DB copy.
- No actual DB write.
- No production DB write.
- No DB schema migration.
- No product import route connection.
- No API/UI import flow.
- No `package.json` script.
- No raw HRC zip access, copy, extraction, or commit.
- No generated artifact JSON.
- No filesystem access.
- No DB access.
- No `process.env`, `Date.now`, or random value usage.

## Approval Contract Relationship

Step 2 defines whether an explicit approval contract is satisfied for a future copied DB rehearsal. Step 3 uses that result as one guard input.

The adapter `buildHrcCopiedDbRehearsalApprovalDecisionFromContract` maps a satisfied approval contract into a guard approval decision. This mapping can mark the future rehearsal as eligible, but it still does not execute a write. The guard output keeps:

- `productionDbWriteAllowed: false`
- `copiedDbWriteAllowed: false`
- `reportFileWriteAllowed: false`
- `actualDbWritePerformed: false`

## Target Policy

The guard rejects:

- `PRODUCTION_DB`
- `REPO_LOCAL_DB`
- `UNKNOWN`

The guard accepts only `COPIED_DB` targets located in:

- `RESTORE_TEST`
- `BACKUP_ROOT_REHEARSAL`

Any copied DB target in `PROJECT_REPO`, `PRODUCTION`, or `UNKNOWN` location fails.

## DB SHA Policy

For a future copied DB write rehearsal, the original DB SHA before and after must both be present and must match. If copied DB write approval is present but the original DB SHA values are missing or different, the guard fails.

The copied DB SHA baseline must also be present before a rehearsal can proceed.

## Required Rehearsal Inputs

The guard requires:

- import preview validation pass
- duplicate/canonical key validation pass
- rollback plan provided
- privacy/path scan pass
- backup manifest available
- source archive DB injection policy acknowledged

Each missing or failed check blocks the copied DB rehearsal.

## Dry-run-only Handling

The guard can return `dryRunOnlyAllowed: true` when:

- target kind and location are safe
- production DB write is not requested
- validation and rehearsal inputs pass
- approval decision allows dry-run
- write flags are false

This state still blocks copied DB write rehearsal and requires explicit copied DB write approval before any future write-capable rehearsal.

## Reject Conditions

The guard rejects:

- production DB target
- repo-local DB target
- unknown target
- copied DB target outside restore-test or backup-root rehearsal locations
- production DB write approval
- missing approval for copied DB write rehearsal
- invalid approval decision
- original DB SHA mismatch
- missing original DB SHA for write rehearsal
- missing copied DB SHA baseline
- missing rollback plan
- failed preview validation
- failed duplicate/canonical key validation
- failed privacy scan
- missing backup manifest
- missing source archive DB injection policy acknowledgment

## Privacy and Path Redaction

The guard output must not expose real local paths, private user tokens, emails, or raw HRC paths. Any private strings in input warnings or blocked reasons are redacted before returning output.
