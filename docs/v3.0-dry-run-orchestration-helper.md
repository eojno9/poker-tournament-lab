# v3.0 Dry-run Command Pure Orchestration Helper

## Purpose

The Step 9 helper fixes the future dry-run import command core as an in-memory orchestration shape. It composes the v3.0 pure helpers created in Steps 2 through 8 and returns one deterministic dry-run result that a later command can display or serialize after a separate approval step.

This step is not a command implementation. It does not add a package script, read the DB, write the DB, write report files, create backups, run restores, call APIs, change UI, or connect product import routes.

## Non-Goals

- No actual import command.
- No `package.json` script.
- No DB read or write.
- No DB schema migration.
- No file write.
- No report JSON generation.
- No backup creation.
- No restore rehearsal.
- No API, UI, server runtime, or product import route connection.
- No raw HRC zip access, copy, extraction, or commit.
- No generated artifact JSON commit.

## Input Shape

`HrcImportDryRunOrchestrationInput` accepts explicit in-memory values only:

- `commandName`
- `timestampIso`
- `branchName`
- `commitHash`
- `dbFileName`
- `dbSha256Before`
- `dbSha256After`
- `previewRows`
- `existingSolutionRows`
- `classificationSummary`
- `safetyChecks`

The helper does not call `Date.now`, random generators, filesystem APIs, DB APIs, HTTP APIs, or command execution. The timestamp is supplied by the caller.

## Output Shape

`HrcImportDryRunOrchestrationResult` includes:

- `version`
- `previewSummary`
- `existingSolutionSnapshot`
- `validationSummary`
- `backupManifest`
- `commandReport`
- `exitCode`
- `status`
- `warnings`
- `nextAction`
- `writeAllowed`
- `dbWriteAllowed`
- `reportFileWriteAllowed`

The write flags are always false. The backup manifest is a preview shape only and does not imply that any backup exists.

## Helper Composition

The orchestrator composes these existing pure helpers:

- `summarizeHrcImportPreviewRows`
- `buildHrcExistingSolutionCanonicalKeySnapshot`
- `validateHrcImportPreviewRows`
- `summarizeHrcImportPreviewValidation`
- `buildHrcImportBackupManifest`
- `buildHrcImportCommandReport`

`previewRows` and `existingSolutionRows` are caller-supplied arrays. The helper does not read existing rows from the product DB.

## Exit Code Policy

- `0`: Dry-run orchestration is OK. Manual review and excluded rows may still exist, but no blocking duplicate, missing key, privacy, or safety issue was detected.
- `1`: Validation blocked by duplicate existing DB key, duplicate batch key, missing canonical key, privacy-blocked candidate, or unexpected DB write allowance.
- `2`: Safety failed because DB SHA256 changed, product import route was enabled, or DB read/write was performed.
- `3`: Privacy/path failure detected, including local path, raw artifact, user token, or email exposure.
- `4`: Invalid command report input.

Manual review candidates are non-fatal in this helper. The v2.9 fixture with 19 `IMPORT_CANDIDATE`, 8 `NEEDS_MANUAL_REVIEW`, and 1 `EXCLUDE` returns exit code `0` when safety, privacy, duplicate, and missing-key checks pass.

## Privacy And Safety Policy

The helper redacts local paths, user tokens, raw HRC folder markers, and email-like values from public result fields. It also treats those values as privacy/path failures.

The result must not expose:

- raw absolute local paths
- user tokens
- email addresses
- raw HRC folder paths
- raw HRC zip content
- generated artifact JSON content

## Write Policy

These values are fixed in every result:

- `writeAllowed: false`
- `dbWriteAllowed: false`
- `reportFileWriteAllowed: false`

Any future write-capable import flow must be a separate step with explicit user approval, copied-DB rehearsal, backup policy, rollback policy, and final verification.
