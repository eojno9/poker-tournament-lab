## Findings

### Fixture and API setup

Large response constants occupy roughly the first 500 lines. A default
`beforeEach` installs common routes, while individual scenarios override
selected routes. This works, but fixture intent and route precedence require
careful reading.

Classification: `v4.2 Should`.

Recommended boundary:

- public-safe fixture constants in a fixture module
- `fulfillJson` and default route installation helpers
- scenario-specific overrides remain beside the scenario

### Navigation and assertions

Navigation is repeated but still readable. A small `openTab` helper can improve
failure messages; a broad workflow DSL would make tests harder to diagnose.

Classification: `v4.2 Could`.

### Failure diagnostics

The test IDs are detailed and most assertions identify the failing panel. The
main diagnostic weakness is broad scenario length, not a lack of selectors.
Keep assertions local and split only when each resulting test has a clear user
intent.

Classification: `maintenance only`.

### Viewport coverage result

The existing Trainer narrow-screen baseline remains, and the HRC scenario now
uses `390 x 900`. It checks document-level width, visible controls, and
table-local horizontal scrolling without screenshot or fixed-pixel geometry
assertions.

Result: `NARROW_VIEWPORT_PASS`.

## Web unit-test baseline

The web suite has 131 tests. Pure models and storage helpers have focused unit
coverage, including safe storage and Korean-first API error mapping. HRC domain
formatting/filtering has five helper tests, while the React screen boundary is
protected mainly by smoke.

For HRC extraction, preserve existing helper tests and smoke coverage. Adding a
new React test framework is not justified.
